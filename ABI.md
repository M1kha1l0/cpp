**Application Binary Interface.**
Формально это не часть стандарта, но почти все компиляторы работают по такому принципу.

Это разбор уже более низкого уровня работы компилятора с полиморфными объектами.

Как мы знаем, любой полиморфный объект содержит в себе таблицу виртуальных функций, а значит даже пустой объект будет весить 8 байт.

```cpp
#include <iostream>

struct Base {
	virtual void f() {}
	virtual ~Base() = default;
	
	int x = 0;
};


int main() {
	std::cout << sizeof(Base) << '\n'; // 8
}
```

Содержание таблицы:
- &Base::type_info - специальный тип для вывода информации о типе объекта
- &Base::f и другие методы класса.

Указатель же на полиморфный объект будет указывать на начало объявления виртуальных методов (в памяти виртуальные методы лежат в порядке объявления)

Выглядит это так:

| &Base::type_info | &base::f() |
| ---------------- | ---------- |
Сам Base выглядит схожим образом:

| ptr (vtable) | x   | ... (поля класса) |
| ------------ | --- | ----------------- |
Не виртуальные методы не влияют на итоговый размер объекта, т.к. лежат в сегменте data и в общем-то не мешают.

Но когда добавляется Derived с виртуальным методом:

```cpp
#include <iostream>

struct Base {
	virtual void f() {}
	virtual ~Base() = default;
	
	int x = 0;
};

struct Derived : Base {
	void f() override {}
	virtual g() {}
	
	int y = 0;
};
```

vtable Derived:

| &Derived::type_info | &Derived::f() | &Derived::g() |
| ------------------- | ------------- | ------------- |

Допустим мы хотим вызвать метод f по ссылке на родительский класс:

```cpp
int main() {
	Derived d;
	Base& b = d;
	b.f();
}
```

Если же мы бы вызывали не виртуальную функцию, то был бы просто call из  функции по указанному адресу. Но в нашем случае нельзя просто дернуть нужный адрес функции, которая будет вызываться, т.к. он известен только в рантайме. Но если запросить явно какую функцию вызывать, то эта задача разрешима на этапе компиляции (логично).
Что же происходит на самом деле:
Чтобы понять какой метод нужно вызвать сначала мы идем по указателю на vtable. К счастью, как объявлены функции и сколько нужно сделать отступов для чтения ее адреса компилятор знает и может прочитать нужный адрес. Таким образом, если под Base& лежит сам Base, то он пойдет по указателю на vtable и оттуда прочитает нужный адрес, если же под Base& лежит Derived, то указатель на vtable уже будет на Derived и т.к. при наследовании сохраняется порядок объявления, то вызывается уже нужная функция от Derived. И все это в компайл тайме.

Отсюда понятно как работает dynamic_cast ([[(!)Различные касты типов]]): идет по указателю на таблицу (который указывает на начало первого метода), для указателя всегда известно сколько нужно отступить влево, чтобы получить указатель на type_info. После получения информации по типу выбирается исход (ошибка или нет).
Уже из этой информации он знает от какого типа был вызван и к какому нужно кастовать, а значит и знает отступы к памяти объекта.

И сразу появляется пример, где dynamic_cast не может кастить вниз: в том случае, если полиморфные объекты появляются не с родительского класса. Тогда при касте к самому верхнему родителю отступом затирается указатель на виртуальную таблицу. Тем самым мы теряем однозначность в сдвиге указателя на следующий объект (не понятно сколько полиморфных объектов было до него, если бы это был например не самый высокий родительский класс).
