# unique_ptr

Реализация и идея данного смарт поинтера очень проста: хочется реализовать идиому RAII для указателей, чтобы не забывать делать освобождение ресурса, объект при завершении жизни сам его освободит. Важное замечание, что владение ресурсами может быть разное и освобождение тоже различное, так, например, не получится освободить ресурс C-файла оператором delete, для него нужен fclose. Для этого в шаблон класса передается структура с определенным в ней функтором (перегруженный оператор круглых скобок). В функторе мы прописываем как именно мы хотим освобождать ресурс данного типа.
Из названия понятно, что у ресурса должен быть один владелец, иначе при множественным владением, будет освобождаться один ресурс, а это UB. Поэтому решение здесь простое - удалить конструктор и оператор копирования, чтобы избежать этой проблемы

 ```cpp
#include <iostream>
#include <memory>

template <typename T>
struct DefaultDeleter {
	void operator()(T* ptr) {
		delete ptr;
	}
};

struct FileDeleter {
	void operator()(FILE* stream) {
		fclose(stream);
	}
};
  
template <typename T, typename Deleter = std::default_delete<T>>
class unique_ptr {
	T* ptr;
	Deleter del;
	
	public:
	unique_ptr(T* ptr, Deleter del)
		: ptr(ptr)
		, del(del)
		{}
	unique_ptr(T* ptr)
		: ptr(ptr)
		{}
	
	// при освобождении ресурса, вызываем переданный функтор
	~unique_ptr() {
		del(ptr);
	}
	
	unique_ptr(const unique_ptr& other) = delete;
	unique_ptr& operator=(const unique_ptr& other) = delete;
	  
	unique_ptr(unique_ptr&& other)
		: ptr(other.ptr)
		, del(std::move(other.del))
		{
		other.ptr = nullptr;
	}
	
	unique_ptr& operator=(unique_ptr&& other) {
		if(this != other) {
			del(ptr);
			ptr = other.ptr;
			del = std::move(other.del);
			other.ptr = nullptr;
		}
		
		return *this;
	}
	
	T& operator*() const {
		return *ptr;
	}
	
	T* operator->() const {
		return ptr;
	}
	
	T* get() const {
		return ptr;
	}
};
 ```

Пример использования:
```cpp
int main() {
	unique_ptr<FILE, FileDeleter> stream(fopen("test.txt", "r"));
	
	char buff[1024];
	fgets(buff, sizeof(buff), stream.get());
	// ручное закрытие файла не требуется
	// функтор самостоятельно освободил ресурс
	// после вызова деструктора 
}
```


# shared_ptr

Идея такая: хочется, чтобы несколько поинтеров могли владеть ресурсом, но при удалении одного поинтера другие копии еще могли владеть ресурсом, а само освобождение ресурса происходило тогда, когда ни один поинтер на него не ссылается. 

Для этого нужно в поле класса добавить счетчик, сколько объектов владеет ресурсом. Наивная идея создать обычное поле переменной - не сработает, при копировании не понятно, что с ним делать (точнее понятно: нужно обновлять счетчик всех владельцев ресурсов, а что если их очень много?). Чуть менее наивная идея - хранить статическое поле, но нет (почти, статическое поле будет общее для всех типов указателей, а что если много разных ресурсов одного типа?).
Вывод: нужно хранить указатель на счетчик.

Базовая реализация требующая доработок:

```cpp
template <typename T>
class shared_ptr {
	T* ptr;
	size_t* count;
	
	public:
	shared_ptr(T* ptr)
		: ptr(ptr)
		, count(new size_t(1))
		{}
	
	shared_ptr(const shared_ptr<T>& other)
		: ptr(other.ptr)
		, count(other.count)
		{
			if(ptr) ++(*count);
		}
	
	  
	
	shared_ptr(shared_ptr<T>&& other)
		: ptr(other.ptr)
		, count(other.count)
		{
			other.ptr = nullptr;
			other.count = nullptr;
		}
	
	shared_ptr& operator=(const shared_ptr& other){
		if(this == other) {
			return *this;
		}
		
		if(ptr && --*count == 0) {
			delete ptr;
			delete count;
		}
		
		ptr = other.ptr;
		count = other.ptr;
		  
		if(ptr) {
			++*count;
		}
		
		return *this;
	}
	
	shared_ptr& operator=(shared_ptr<T>&& other){
		ptr = std::move(other.ptr);
		count = std::move(other.count);
	}
	
	~shared_ptr() {
		if(ptr && --(*count) == 0) {
			delete count;
			delete ptr;
		}
	}
	
	T* operator->() const {
		return ptr;
	} 
	
	T& operator*() const {
		return *ptr;
	}
	
	T* get() const {
		return ptr;
	}
};
```

На самом деле у shared_ptr тоже есть свой делитер, но он не является частью объекта, поэтому вес такого поинтера 16 байт.
Но и здесь не все так просто, есть функции создающие экземпляр класса по объектам - make_shared и make_unique. И нужно поддерживать оба варианта создания объекта (по объекту и по указателю).
Для этого введен специальный объект ControlBlock - он содержит счетчик ссылок (на самом деле 2 счетчика: ссылки сильной связи и слабой связи, об этом в weak_ptr) и пользовательский делитер. Само поле является указателем от чего достигается размер в 16 байт.
Создается он в 3 конструкторах: от объекта (std::make_shared), от unique_ptr, от указателя.

Но нужно помнить о проблеме конструктора от указателя:

```cpp
#include <memory>

int main() {
	int *ptr = new int(10);
	std::shared_ptr<int> p1(ptr);
	std::shared_ptr<int> p2(ptr);
}
```

В данном примере конструктор будет вызван 2 раза и создастся 2 разных ControlBlock и при удалении одного из поинтеров будет UB.
Да, shared_ptr во многом спасает, но от явного желания выстрелить себе в ногу - нет.
Решение данной проблемы будет вызов std::make_shared

Теперь поговорим про **std::make_shared** и **std::make_unique**. Эти функции позволяют создать соответствующий смарт поинтер от объекта, а не от сырого указателя. Функция внутри себя самостоятельно аллоцирует память (и на объект и на счетчик), вмешательство пользователя не требуется. Это хорошая практика по кодстайлу.

make_unique придумано по следующей причине:

```cpp
foo(unique_ptr<int>(new int(10)), g());
```

Допустим есть такая функция. Изначально вычисление аргументов функции было неупорядоченное, т.е. вполне себе возможно поведение компилятора, что сначала вызовется оператор new, потом вычисляется функция g(), и только потом вызывается конструктор смарт поинтера. Допустим сначала вызвался оператор new, а потом стала считаться g(). Внезапно g() может выкинуть исключение, а так как конструктор поинтера не вызван, то ресурсы некому освобождать, получаем утечку.
В то же время make_unique это функция и выполняется полностью, так что такая ситуация исключается. (Но с c++17 такой проблемы уже нет, но считается плохим тоном использовать сырые указатели).

Изначально make_shared придумали, чтобы избежать двойного вызова оператора new (это долго). Если пользоваться конструктором по указателю, то приходится вызывать 2 раза, но зачем, если можно за один раз аллоцировать объект и положить рядом счетчик. Для этого создается специальная структура содержащая и счетчик и объект (ControlBlock)
Как работает:

```cpp
template <typename T, typename... Args>
unique_ptr<T> make_unique(Args&&... args) {
	return unique_ptr<T>(new T(std::forward<Args>(args)...));
}

// friend функция класса
template <typename T, typename... Args>
shared_ptr<T> make_shared(Args&&... args) {
	auto* p = new typename shared_ptr<T>::ControlBlock{T(std::forward<Args>(args)...), 1};
	return shared_ptr<T>(p);
}
```

У shared_ptr дополнительно появляется приватная структура внутри класса ControlBlock, которая отвечает за хранение блока объекта и счетчика. Но нужно поддерживать как конструктор из указателя, так и из ControlBlock и чтобы итоговый вес был всего 16 байт (2 указателя).

Примерно так:

```cpp
template <typename T, typename Deleter = std::default_delete<T>>

class shared_ptr {
	T* ptr;
	Deleter del;
	size_t* count;
	
	struct ControlBlock {
		T value;
		size_t count;
	};
	
	shared_ptr(ControlBlock* cp) { /* implement me */ }
	
	template <typename U, typename... Args>
	friend shared_ptr<U> make_shared(Args&&...);
	
	public:
	shared_ptr(T* ptr)
		: ptr(ptr)
		, count(new size_t(1))
		{}
	
	shared_ptr(const shared_ptr& other)
		: ptr(other.ptr)
		, count(other.count)
		{
			++*count;
		}
	
	shared_ptr(shared_ptr&& other)
		: ptr(std::move(other.ptr))
		, count(std::move(other.count))
		{}
	
	shared_ptr& operator=(const shared_ptr& other){
		ptr = other.ptr;
		count = other.count;
		++*count;
	}
	
	shared_ptr& operator=(shared_ptr&& other){
		ptr = std::move(other.ptr);
		count = std::move(other.count);
	}
	
	~shared_ptr() {
		if(!count) return;
		--*count;
		
		if(!*count) {
			delete count;
			delete ptr;
		}
	}
	
	T* operator->() const {
		return ptr;
	} 
	
	T& operator*() const {
		return *ptr;
	}
	
	T* get() const {
		return ptr;
	}
};
```

# weak_ptr

Хотим полностью отказаться от сырых указателей и пишем только смарт поинтерами.
Допустим у нас есть дерево красно черное, у него могут быть циклические зависимости, тогда при удалении корня память никуда не освободится. Для таких случаев сделан weak_ptr.
По сути своей weak_ptr это слабая ссылка на объект, который не учитывается при подсчете объектов-владельцев ресурса. (Это в целом определение слабой ссылки в C++).
Таким образом, чтобы избежать такого момента нужно, чтобы в цикле была хотя бы одна слабая ссылка. Например, от родителей к детям делаем обычный shared_ptr, а в обратную сторону weak_ptr, тогда решается проблема с удалением корня.

У weak_ptr нет операторов разыменования и обращения по стрелке, а также нельзя получить указатель на объект из weak_ptr, т.к. это будет UB. weak_ptr не владеет ресурсом объекта, а значит, что он может ссылаться на особожденый ресурс, поэтому эти операторы запрещены.
Проверить ссылается ли на освобожденный ресурс можно с помощью метода `expired()`.
Но как тогда обращаться к объекту, если есть только weak_ptr? Для этого есть метод `lock()`. Он автоматически проверит, не является ли текущий указатель "висячим" и вернет сконструированный shared_ptr, если же висячий указатель, то вернет shared_ptr от nullptr.
Есть и второй способ завладеть ресурсом, на который ссылается weak_ptr, - вызов конструктора shared_ptr. Но есть нюанс, если указатель weak_ptr "висячий", то конструктор выкинет исключение, что не есть хорошо.

Важно, что weak_ptr, если ссылается на объект, сохраняет ControlBlock shared_ptr, на который ссылается (для этого нужен счетчик слабых ссылок), чтобы понять, что объект является висячим.
