# Структуры
Объявление структуры

```cpp
struct S {
	int x;
	int y;
};
```

Теперь я могу создавать объекты типа **S** и как-то с ними работать.
Чтобы обращаться к полям структуры (их может  быть несколько) используется оператор **точка**, который невозможно переопределить.

```cpp
struct S {
	int x;
	int y;
};

int main() {
	S s; // создание объекта структуры выглядит как обычное объявление переменной
	S.x; // обращение к полю
}
```

Также для структуры можно инициализировать значение полей при создании экземпляра этой структуры, а также менять поля:

```cpp
#include <iostream>

struct S {
	int x = 0;
	int y;
};

int main() {
	S s;
	std::cout << s.x << '\n'; // 0
	std::cout << s.y << '\n'; // UB
}
```

Также поля структуры можно инициализировать при объявлении через фигурные скобки - это называется **агрегатной инициализацией**:

```cpp
#include <iostream>

struct S {
	int x = 0;
	int y;
};

int main() {
	S s{1, 2};
	std::cout << s.x << '\n'; // 1
	std::cout << s.y << '\n'; // 2
}
```

Если же передать значений меньше, чем полей, то они проинициализируются, как указано в структуре (если указано).
Если же передать больше, чем нужно - ошибка компиляции.
**Важно**: агрегатная инициализация доступна, если отсутствует конструктор и поля структуры публичные.

Внутри структур (и не только структур, можно в любых скоупах внутри) можно объявлять другие структуры (даже анонимные) и сразу по ним создавать объекты.

```cpp
struct S {
	int x;
	int y;
	
	struct SS {
		char c;
	} a; // объявляется поле типа SS под именем a
	
	struct {
		int64_t l;
	} a2; // обявление поля анонимной структуры
};

int main() {
	S::SS a; // создание экземпляра структуры объявленной внутри
	// к анонимной структуре обратиться нельзя
	S.a.c; // можно обратиться к полям внутренней структуры 
}
```

Но важно понимать, только объявление не создает экземпляр структуры.

## Объявления методов
Функции объявленные внутри структуры называются методами и имеют доступ изменять поля структуры, даже если они приватные.
Внутри структуры запрещено объявлять новые пространства имен, но зато можно использовать using, тогда область видимости для методов расширяется во всех структуре (но **using для переменных нельзя**).
Обращение к внутренним методам также как и к полям через оператор точка.
Пример:

```cpp
#include <iostream>

struct S {
	int x = 0;
	double d;
	
	// так делать неправильно, но для примера
	using namespace std;
	
	void f(int y) {
		// std теперь не нужно
		cout << x + y;
	}
};

int main() {
	S s{1, 2};
	s.f(5); // выведет 6
}
```

Полезное свойство структур: все методы внутри структуры считаются объявленными и их можно использовать в другом методе до объявления.

Как быть с тем, что я хочу написать определение метода структуры в другом месте?
Нужно объявить метод внутри структуры и объявить принадлежность к классу через двойное двоеточие, в начале написав, к какому именно классу.

```cpp
struct S {
	int x = 0;
	double d;
	
	void f(int y);
};

void S::f(int y) {
	std::cout << x + y;
}
```

### Специальные методы
Специальными методами называются методы и операторы, которые определяют: конструктор, деструктор, оператор присваивания, оператор копирования, конструктор перемещения.
Если явно определен хоть один из специальных методов, то необходимо явно удалить или определить остальные.
## this
Это специальное ключевое слово, которое позволяет обратиться к самому себе  по указателю.
Это нужно например, если хочется вернуть копию объекта (указатель нужно разыменовать), а также если хочется явно указать, что меняется поле данной структуры.

```cpp
struct S {
	int x = 0;
	double d;

	void g(int x) {
		// обозначаем явно, что меняем поле текущего объекта
		this->x = x;
	}
};
```

Для указателей тоже есть оператор обращения к полю - это стрелочка (работает аналогично точке)
## Размер структуры и представление в памяти
Стандарт гарантирует, что в памяти поля структуры лежат в порядке их объявления. Это сделано специально, чтобы можно было делать reinterpret_cast из структуры в структуру.

Рассмотрим размер на примере нескольких структур

```cpp
// 8 bytes
struct S1 {
	int x;
	int y;
};

// 16 bytes, not 12
struct S2 {
	int i;
	double d;
};

// 16 bytes
struct S3 {
	int i1;
	int i2;
	double d;
};

// 24 bytes, not 16
struct S4 {
	int i;
	double d;
	char c;
};

// 16 bytes
struct S5 {
	int i;
	char c;
	double d;
};

struct S6 {};
```

Может показаться, что итоговый размер структуры это просто сумма размеров полей. Но нет, это с точностью до выравнивания
		Да, для структуры с одинаковыми полями, общий размер будет действительно суммой, т.к. выравнивать там нечего.
Выравнивание делается с целью разместить массив структур непрерывным куском памяти. Выравнивание для структуры **S2** будет 8, т.к. компилятор кладет int по адресам кратным 4 (размер 4 байта), а double по адресам кратным 8 (размер 8 байт), поэтому для int будет зарезервированно место 8 байт, а не 4. Если же поля одинаковые, то никаких проблем с размещением структур в массив не возникает.
Для структуры **S3** тоже ничего выравнивать не нужно, т.к. 2 подряд int занимают столько же, сколько и double.
**Отсюда общее правило**: размер структуры зависит от общего остатка при делении на максимальный размер поля (выравнивание минимизируется), а также зависит от порядка переменных (минимум выравнивания тоже зависит от порядка).

Разберем примеры 4 и 5:
В 4 примере 24 байта, т.к. гарантируется порядок переменных в памяти, а значит нужно выровнять int и char, оба добавят пустых мест.
В 5 примере 16 байт, т.к. размер char + int меньше 8 байт и они находятся в порядке, что нужно выровнять только их сумму до 8, следовательно 16 байт.

В примере 6 показана пустая структура, но она занимает не 0 байт, а 1 байт. Это сделано, чтобы можно было отличать пустые структуры друг от друга при создании массива, например. ^717f8a

## union
Это объединение нескольких полей. Под union выделяется память под максимальный размер  поля, а далее каждое поле записывается именно в этот блок памяти, который выделили.
Но важно помнить, что выравнивание распространяется и на union.

Union выглядит так:

```cpp
#include <iostream>

union {
	uint32_t u32;
	uint8_t bytes[4];
} u;

int main() {
	u.u32 = 0xaabbccdd;
	std::cout << u.u32 << '\n';
	std::cout << u.bytes[0] << '\n'; // UB
}
```

Если записать какое-то значение в одно поле union, а потом попытаться прочитать значение по другому полю, то произойдет неявный reinterpret_cast к типу поля, к которому обращаемся, и это формально UB.
# Классы
Вообще говоря, структура и класс неотличимы за исключением того, что в структуре по умолчанию все поля публичные, а в классе приватные (разные модификаторы доступа по умолчанию). 
Модификаторы доступа также доступны и для стурктур
## Модификаторы доступа
Позволяют определить, можно ли извне обращаться к полям класса или нет.

### private
Обозначает, что только члены класса могут обращаться к полям класса и к внутренним объявлениям классов и структур. При объявлении все классы имеют такой модификатор доступа для своих полей и методов.

```cpp
class C {
	// по умолчанию private
	int x;
	// но можно явно указать
	private:
		int y;
};
```

При обращении к private полям и методам класса на этапе компиляции выкинет с ошибкой доступа. Обращение определено, но доступа нет.

НО private поля (только переменные) можно вытащить из класса используя reinterpret_cast:

```cpp
#include <iostream>

class C {
	// по умолчанию private
	int x = 10;
};

int main() {
	C c;
	std::cout << (int&)c << '\n'; // 10
}
```

C-style-cast сделает все, чтобы вытащить это значение.

#### friend методы
Данные методы позволяют работать с приватными полями класса, не являясь при этом членом этого класса.
Для этого необходимо с помощью ключевого слова **friend** в классе объявить это функцию, чтобы выдать доступ.
Пример:

```cpp
#include <iostream>

class C {
	// по умолчанию private
	int x = 10;
	
	friend void f(const C&, int);
};

void f(const C& c, int y) {
	std::cout << c.x + y << '\n';
}

int main() {
	C c;
	std::cout << (int&)c << '\n'; // 10
}
```

Формально никто не запрещает определять функцию внутри класса и она все равно будет в глобальном скоупе, но так делать не надо.

Также можно сделать **friend-ами** классы и структуры, тогда как и для **friend** методов обращение к приватным полям из **friend** структуры или класса будет разрешено.

Но также важное уточнение, что friend несимметричен и нетранзитивен.
### public
Обозначает, что кто угодно может обращаться к полям класса. При объявлении все структуры имеют такой модификатор доступа для своих полей и методов, для классов нужно это явно указывать.

```cpp
class C {
	public:
		int y;
		
		void f();
		
	private:
		int x;
};

int main() {
	C c;
	c.y; // ошибки нет
	c.f(); // тоже нет ошибки
	c.x; // ошибка доступа
}
```


### protected
Чуть более сложная структура созданная для наследования.
Protected позволяет дать доступ кроме членов класса и friend-ов еще и наследникам родительского класса.

## Конструкторы

^c66cea

Рассмотрим на примере класса комплексных чисел.
Используются для создания объекта по некоторым параметрам. Это метод класса, у которого нет возвращаемого значения, а название метода совпадает с названием класса. В аргументах ожидаемые параметры для создания объекта.

```cpp
class Complex {
	double re = 0.0;
	double im = 0.0;
	
	// конструктор по вещественной части
	public:
		Complex(double real) {
			re = real;
		}
};

int main() {
	Complex c(5.1); // прямая инициализация с вызовом конструктора
	Complex c2 = 6.0; // здесь тоже вызывается конструктор, это не присваивание
	Complex c3{7.0}; // тоже выызов конструктора 
}
```

Но такой конструктор не самый эффективный, когда поля являются сложными классами с нетривиальным конструктором. Лучше определить так:

```cpp
class Complex {
	double re = 0.0;
	double im = 0.0;
	
	public:
		Complex(double re): re(re) {}
};
```

Фактически выполняется одинаковое действие, но до конструктора и на 1 вызов меньше. 

Определим еще один конструктор

```cpp
class Complex {
	double re = 0.0;
	double im = 0.0;
	
	public:
		Complex(double real): re(real) {}
		Complex(double re, double im): re(re), im(im) {}
};
```

Важно понимать, что по стандарту такая инициализация должна быть в порядке объявления полей, на случай, если поля зависят друг от друга.
Также не стоит писать внутри фигурных скобок тривиальные инициализации.

Заметим что `Complex() {}` это конструктор по умолчанию и сам по себе компилятор генерирует его самостоятельно, если нет каких-либо других конструкторов.
Иначе нужно его определить вручную.
Но можно попросить компилятор доопределить конструктор по умолчанию, если сказать ему `Complex() = default;` (объявляется явно, но неявно определяется). Такой конструктор проинициализирует поля значениями по умолчанию, если они есть.
Но бывают ситуации, когда конструктор по умолчанию сгенерировать автоматически невозможно.
В случаях, когда поля класса содержат ссылки и константы, которые не имеют значения по умолчанию, а также классы не имеющие тривиальных конструкторов.
### initializer list

^05cb38
Это когда конструктор принимает в себя {} с произвольным количеством аргументов.
В векторе это так: `std::vector<int> v = {1, 2, 3, 4, 5}`
Напишем что-то подобное:

```cpp
class String {
	char* arr = nullptr;
	size_t sz = 0;
	size_t cap = 0;
	
	public:
		String() {} // default
		String(size_t n, char c): arr(new char[n + 1]), sz(n), cap(n + 1) {
			std::fill(arr, arr + sz, c);
			// это то же что и memset(arr, c, n);
			arr[sz] = '\0';
		}
		// вот этот конструктор позволяет передавать много аргументов через {}
		String(std::initializer_list<char> list)
				: arr(new char[list.size() + 1])
				, sz(list.size())
				, cap(sz + 1) {
			std::copy(list.begin(), list.end(), arr);
			arr[sz] = '\0';
		}
};
```

По сути своей этот список переменных это захардкоженый тип в компилятор, который под капотом позволяет делать такие вещи.

Теперь если попытаться в main вызвать конструктор из 2-х аргументов в фигурных скобках, то вызывается **initializer_list**, всегда, если это возможно (если невозможно, то вызовется обычный конструктор, а если и это не сработало, то агрегатная инициализация).

```cpp
int main() {
	String s{'a', 'b', 'c'}; // initializer_list конструктор
	String s1{2, 'a'}; // тоже initializer_list конструктор
	String s2(2, 'a'); // обычный конструктор
}
```

## Деструкторы
Функция обратная конструктору, вызывается перед удалением объекта.
Для тривиальных типов деструктор ничего не делает (как в примере с классом Complex), а вот если есть нетривиальный конструктор, то деструктор тоже должен быть нетривиальным и освобождать какие-нибудь ресурсы (например освобождение дин. памяти).
Посмотрим на код String, который написали в разделе [[#^05cb38|initializer list]]. 

```cpp
class String {
	char* arr = nullptr;
	size_t sz = 0;
	size_t cap = 0;
	
	public:
		String() {}
		String(size_t n, char c): arr(new char[n + 1]), sz(n), cap(n + 1) {
			std::fill(arr, arr + sz, c);
			arr[sz] = '\0';
		}
		
		String(std::initializer_list<char> list)
				: arr(new char[list.size() + 1])
				, sz(list.size())
				, cap(sz + 1) {
			std::copy(list.begin(), list.end(), arr);
			arr[sz] = '\0';
		}
		
		// нетривиальный деструктор
		~String() {
			delete[] arr;
		}
};
```

Деструктор это такой же метод и его можно вызвать явно (через точку).
Если бы для String не был бы написан деструктор, то была бы утечка памяти.

Вообще, если внутри класса были бы нетривиальные поля со своими деструкторами, то при вызове из текущего класса деструктора, сначала срабатывает он, а потом уже деструктор для полей.

# ООП (Объектно Ориентированное Программирование)
Одна из парадигм программирования на C++.
Прадигма программирования, которая позволяет определять свои типы данных.
## Абстракция
Это классы с хотя бы одной полностью виртуальной функцией.

```cpp
struct Shape {
	// полностью виртуальная функция
	virtual double area() const = 0;
	virtual ~Shape() = default;
};
```

Если после функции стоит 0, это значит что определение функции в этом классе не ожидается, но во всех наследниках должно быть ее определение.

Невозможно создать объект абстрактного класса, можно только указатель или ссылку с типом абстрактного класса.

```cpp
struct Shape {
	// полностью виртуальная функция
	virtual double area() const = 0;
	virtual ~Shape() = default;
};

struct Square {
	double a;
	Square(double a): a(a) {}
	
	double area() const override {
		return a * a;
	}
};

int main() {
	Shape a; // ошибка!
	
	Square sq(1.0);
	Shape& s = sq;
	s.area(); // вызовется area у square. Это полиморфный объект
}
```

Понятно, что можно создать вектор указателей Shape и записывать туда объекты наследников и вызываться area будет у дочерних классов.

Позволяет защитить данные от изменения.

## Наследование
Основной принцип это наследование каких-то методов из родительского класса. Т.е. родительский класс это общий случай, а дочерний это частный случай.

Пример синтаксиса:
```cpp
struct Base {
	int x;
	void f() {}
};

// Derived наследник Base
struct Derived : Base {
	int y;
	void g() {}
};
```

В данном случае Derived унаследует все поля и функции Base (т.е. в Derived будет и x b y, и f и g, а в Base только f и x).

Важно, что friend функции не наследуются.

Но само наследование тоже имеет модификаторы доступа и для классов по умолчанию private наследование, а у структур public. Модификаторы доступа наследования применимы только к наследуемым элементам класса (только public и protected, private элементы наследуются, но никто не имеет к ним доступа). 
Но что вообще модификатор доступа наследования:
-  private наследование это когда методы и поля, которые унаследовал класс от родителя может использовать только член дочернего класса или его friend.
- public наследования это когда из всех скоупов известно, что этот класс наследник родительского класса (т.е. унаследованные методы и поля публичные).
-  protected наследование это когда только член класса, friend функции и наследники могут пользоваться всем унаследованным.

Упрощенно: для всех унаследованных полей и методов ставится модификатор доступа, который мы указываем в объявлении наследования, он изначально был доступен (если не указываем, то значение по умолчанию).

Наследование методов с одинаковым именем: расставляется приоритет - частное приоритетнее общего. Более того, функция в наследнике затмевает  родительскую функцию и при вызове попросту не рассматривает унаследованные методы с таким же именем, т.е. перегрузка не происходит.

Например:

```cpp
#include <iostream>

struct Base {
	int x;
	void f() {
		std::cout << 1;
	}
	
	void g(int) {
		std::cout << 1;
	}
	
	void h(int) {
		std::cout << 1;
	}
};

// Derived наследник Base
struct Derived : Base {
	int y;
	void f() {
		std::cout << 2;
	}
	
	void g(double) {
		std::cout << 2;
	}
	
	void h() {
		std::cout << 2;
	}
};

int main() {
	Derived d;
	d.f(); // выведет 2
	d.g(0); // тоже выведет 2
	d.h(0); // ошибка компиляции, в Derived нет метода с такими аргументами
}
```

Но определение родительской функции все еще лежит в классе наследника и поэтому к ней все-таки можно обратиться через двойное двоеточие.

```cpp
int main() {
	d.Base::g(0); // выведет 1
}
```

Важно ! Модификаторы доступа не влияют на определение функции, всегда будет вызываться функция текущего класса, даже если это ошибка компиляции.

Но сделать так, чтобы перегрузка работала можно через using:

```cpp
// Derived наследник Base
struct Derived : Base {
	// добавляем в область видимости Derived
	using Base::g;

	int y;
	void f() {
		std::cout << 2;
	}
	
	void g(double) {
		std::cout << 2;
	}
	
	void h() {
		std::cout << 2;
	}
};
```

Более того, using позволяет сделать из protected метода public метод. Наследование этого не запрещает, метод унаследовался, но он не в области видимости дочернего класса. С private методами это не прокатит, тут уже наследование запрещает к нему обращаться.
Но тут важен и с каким модификатором доступа добавляется юзингом в область видимости, все не публичное отменит перегрузку.

Т.е. видимость и доступность имеет разную логику. Сначала набираются видимые, из них выбирается лучший кандидат, а после проверятся доступ.
В целом это касается всего содержания класса. 

Наследование конструкторов. При создании объекта дочернего класса сначала вызывается конструктор родителя, а потом конструктор дочернего класса. Т.е. если есть только пользовательские конструкторы у родителя и у ребенка, то создание объекта дочернего класса невозможно ибо нечем инициализировать поля родителя (или придется явно в конструкторе дочернего класса прописывать чем инициализировать поля родителя). 
Т.е. общий порядок инициализации дочернего класса: сначала инициализируются поля родителя (вызываются конструкторы классов в полях, если есть), затем наследник.

Деструктор при наследовании работает в обратном порядке: сначала дочерний класс, потом родительский и поля родительского класса.

Конструктор копирования не наследуется.

Самая важное свойство наследования: везде где ожидается родитель, можно подсунуть дочерний класс. Если передавать объект, то неявно создастся конструктор родительского класса из дочернего и достроенные поля и методы отбросит. В обратную сторону так не работает, не из чего доопределять родительский класс. 
### Как расположены в памяти наследники
У наследников суммируются все поля и выравниваются по принципу, который обсудили в структурах.
Все поля записываются в конец и лежат таком же порядке. Но если это пустой класс (т.е. есть только методы, но нет полей, говорили [[Структуры и классы#^717f8a|здесь]]), то он ничего дополнительно не занимает, хоть и имеет размер 1 байт.

### Множественное наследование
Класс может наследоваться от нескольких классов сразу с различными модификаторами наследования.
В памяти они лежат в порядке объявления наследования: сначала родители в порядке объявления, а потом сам дочерний класс.

```cpp
struct A {
	int a;
};

struct B {
	int b;
}

struct C : A, B {
	int c;
}
```

При множественном наследовании в преобразованиях типов происходит сдвиг указателя, на тот тип родителя, к которому передаем.

Проблемы множественного наследования:
-  Если методы родителей называются одинаково, то компилятор делает перегрузку для этих методов. Если есть одинаковые поля, то при обращении к ним будет ошибка компиляции, а так они унаследуются.
-  Проблема ромбовидного наследования. Это когда два или более родителей дочернего класса являются дочерними классами одного главного класса.

```cpp
// ромбовидное наследование
struct A {
	int a;
};

struct B : A {
	int B;
};

struct C : A {
	int c;
};

struct D : B, C {
	int d;
};
```

Разберем ромбовидное наследование.
Посмотрим как это лежит в памяти: у B и C в памяти лежит по экземпляру A. при наследовании D эти участки памяти копируются, т.е. будет 2 копии класса A в разных местах (в начале и через класс B). 
При попытке обратиться к переменной из класса A, будет гарантированная ошибка компиляции из-за двойного определения, но это можно обойти: `d.C::a`, но не `d.A::a` - это уже ошибка компиляции, ибо A повторяется, не понятно к какому обращаться.
Кроме того каст к классу A невозможен, будет ошибка компиляции из-за двух копий A (не понятно к какому именно). Но опять же есть обходной вариант: сначала кастить к B или C и от них уже к A. И наоборот каст тоже невозможен, не понятно через какого родителя кастить.
Также обстоят дела и с методами класса A. В зависимости от полей класса они могут работать по-разному, следовательно ошибка  компиляции.

### Как исправить проблемы ромбовидного наследования?
Если очень понадобилась такая структура, то можно наследоваться от класса A виртуально, тогда исправляется двойное копирование класса A в памяти класса D

```cpp
struct A {
	int a;
};

struct B : virtual A {
	int B;
};

struct C : virtual A {
	int c;
};

struct D : B, C {
	int d;
};
```

Теперь все работает и компилируется и доступны все плюшки наследования.

#### Проблемы такого подхода (!)
Как это лежит в памяти? Хранится указатель на класс А и где нужно подставляется. Важно, что указатель тянется за собой при всех наследованиях от начала виртуального наследования.
Также виртуальные наследования образуют виртуальную таблицу (vtable) наследований, которая хранится в сегме нте памяти Data.
## Полиморфизм
```cpp
#include <iostream>

struct Base {
	void f() {
		std::cout << 1;
	}
};

struct Derived : Base {
	void f() {
		std::cout << 2;
	}
};

int main() {
	Derived d;
	Base& b = d;
	b.f(); // 1
}
```

Т.к. C++ статически типизированный язык, то в ссылке на Base вызывается метод из Base, т.к. изначальный тип Base.

Чтобы вызывался метод наследников, нужно добавить ключевое слово **virtual**:

```cpp
#include <iostream>

struct Base {
	virtual void f() {
		std::cout << 1;
	}
};

struct Derived : Base {
	void f() {
		std::cout << 2;
	}
};

int main() {
	Derived d;
	Base& b = d;
	b.f(); // 2
}
```

Получаем, что виртуальная функция это такая функция, которая при вызове от дочернего объекта по ссылке на родителя, она сработает как метод из дочернего класса (то же самое и от указателя). Но если это копия, то вызывается от родителя.
Важно, чтобы у виртуальной  и у функции наследника совпадали сигнатуры функций полностью, иначе будет вызываться родительский  метод.

Таким образом получим: класс, у которого есть хотя бы одна виртуальная функция, называется полиморфным. Также и класс, который унаследовал виртуальную функцию, является полиморфным.

Деструктор у полиморфных классов тоже должен быть виртуальным иначе, при создании полиморфного объекта и при его последующем удалении, дочерний класс, под полиморфной оберткой не вызовет свой деструктор.

Если же очень хочется переписать сигнатуру виртуальной функции, то нужно явно указать флаг **override** (переписать определение).
Чтобы запретить своим наследникам определять виртуальную функцию пишут ключевое слово **final**.

Важно понимать, что виртуальные функции это рантайм явление и поэтому в независимости от модификатора доступа она будет вызываться. Модификатор доступа зависит только от родительского класса виртуальной функции, если вызвать от приватного модификатора, то это ошибка компиляции уже будет.

### vtable
Это таблица поиска функций используемая для разрешения вызовов виртуальных функций в рантайме.
Каждому классу имеющему виртуальные функции предоставляется собственная таблица. Она представляет собой статический массив указателей на самую производную виртуальную функцию доступную для данного класса. Если виртуальная функция не определена в дочернем классе, то сохраняется указатель на родительскую виртуальную функцию. 
Компилятор скрыто добавляет указатель на таблицу как член в базовый класс.

## Перегрузка операторов

Для всех пользовательских классов C++ дает возможность перегрузить все доступные операторы. При этом приоритеты и порядок вычислений остается неизменным всегда, пользователь не может их определять. Также недоступно доопределение операторов (введение новых).

```C++
struct Complex {
	double re = 0.0;
	double im = 0.0;
	
	Complex(double re): re(re) {}
	Complex(double re, double im): re(re), im(im) {}
	
	// в такой реализации есть проблема
	Complex operator+(const Complex& other) const {
		return Complex(re + other.re, im + other.im);
	}
};
```

Проблема в том, что есть ситуация, когда нужно сложить Complex и double. Нет оператора для сложения double+Complex, а оператор, что мы определили подходит только для Complex+double.
Почему так: как мы знаем аргумент Complex является членом класса. Если же левый аргумент является членом класса, то компилятор найдет перегруженный оператор в определении класса, в котором этот объект является членом. Если же левый аргумент не является членом класса, то компилятор не будет перебирать все касты левого аргумента, чтобы вызвать нужный оператор, и выкинет ошибку.
Поэтому определение операторов выносится из класса, чтобы сделать перегрузку оператора симметричной.
Правильная реализация выглядит так:

```C++
struct Complex {
	double re = 0.0;
	double im = 0.0;
		
	Complex(double re): re(re) {}
	Complex(double re, double im): re(re), im(im) {}
};

Complex operator+(const Complex& lhs, const Complex& rhs) {
	return Complex(lhs.re + rhs.re, lhs.im + rhs.im);
}
```

Теперь хотим определить оператор +=. Этот оператор должен являться членом класса, потому как он модифицирует неконстантный член класса.
Наивная реализация:
```C++
struct Complex {
	double re = 0.0;
	double im = 0.0;
		
	Complex(double re): re(re) {}
	Complex(double re, double im): re(re), im(im) {}
	
	// очень плохая реализация
	Complex& operator+=(const Complex& other) {
		*this = *this + other;
		return *this;
	}
};

Complex operator+(const Complex& lhs, const Complex& rhs) {
	return Complex(lhs.re + rhs.re, lhs.im + rhs.im);
}
```

Такая реализация неэффективна из-за использования операторов:
```C++
Complex& operator+=(const Complex& other) {
	// здесь вызывается оператор копирования (=)
	// и конструктор класса (+)
	// итого сложность 2n
	*this = *this + other;
	return *this;
}
```


Теперь поговорим про эффективную реализацию.
На самом деле += это очень быстрая операция, которая при небольших размерах правого аргумента, выполняется за O(1). Оператор + будет неизбежно вызывать конструктор класса, а значит выполняется за O(n). Поэтому будем определять оператор + через +=:

```C++
struct Complex {
	double re = 0.0;
	double im = 0.0;
		
	Complex(double re): re(re) {}
	Complex(double re, double im): re(re), im(im) {}
	
	Complex& operator+=(const Complex& other) {
		// теперь работает за O(1)
		re += other.re;
		im += other.im;
		return *this;
	}
};

Complex operator+(const Complex& lhs, const Complex& rhs) {
	Complex result = lhs;
	result += rhs;
	return result;
}
```

Но эта реализация все еще не идеальна. В такой реализации возможна такая ситуация вызова оператора:

```C++
int main() {
	Complex a(1.0);
	Complex b(2.0);
	Complex c(3.0);
	
	a + b = c;
}
```

Это работает, потому как переопределение rvalue значений по умолчанию не запрещено для пользовательских классов.
	На пальцах: оператор в текущей реализации возвращает неконстантный экземпляр класса Complex, а для неконстантных типов разрешен оператор присваивания.
Это можно исправить двумя способами

Изменить тип возвращаемого значения:
```C++
const Complex operator+(const Complex& lhs, const Complex& rhs) {
	Complex result = lhs;
	result += rhs;
	return result;
}
```

Это исправляет ситуацию, но так делать это плохая практика.
Правильно реализовать это значит объявить, что оператор + применим только к lvalue значению.
Для этого необходимо изменить оператор присваивания:

```C++
struct Complex {
	double re = 0.0;
	double im = 0.0;
	
	Complex& operator=(const Complex& other) & {
	
	}
		
	Complex(double re): re(re) {}
	Complex(double re, double im): re(re), im(im) {}
	
	Complex& operator+=(const Complex& other) {
		re += other.re;
		im += other.im;
		return *this;
	}
};
```

Заметим, что в конце определения сигнатуры функции стоит **&**. Это явное обозначение применимости для **lvalue**, если бы стояло **&&**, то только для **rvalue**. (Начиная со стандарта C++11).

## Перегрузка операторов ввода и вывода

Для перегрузок такого типа операторов всегда передается еще ссылка на поток ввода/вывода. Потому что, мы что-от хотим читать из цепочки и что-то выводить в цепочку, к тому же не хочется тратить время на копирование. 

Определяются они так:

```C++
struct Complex {
	double re = 0.0;
	double im = 0.0;
	
	Complex(double re): re(re) {}
	Complex(double re, double im): re(re), im(im) {}
	
	Complex& operator+=(const Complex& other) {
		re += other.re;
		im += other.im;
		return *this;
	}
	
	friend std::istream& operator>>(std::istream& in, Complex& copm);
};

Complex operator+(const Complex& lhs, const Complex& rhs) {
	Complex result = lhs;
	result += rhs;
	return result;
}  

std::ostream& operator<<(std::ostream& out, const Complex& comp) {
	out << comp.re << " + i*" << comp.im << '\n';
	return out;
}

std::istream& operator>>(std::istream& in, Complex& comp) {
	in >> comp.re >> comp.im;
	return in;
}
```

Заметим, что у класса **Complex** нет внутренних функций для взаимодействий с внутренними полями, поэтому оператор ввода имеет смысл сделать **friend** оператором. Это, пожалуй, единственный случай, когда стоит применять данный модификатор функции.

## Перегрузка операторов сравнения
Операторы сравнения принимают две константные ссылки на объекты классах и возвращает значение типа bool.

Определим оператор **<**:
```C++
bool operator<(const Complex& lhs, const Complex& rhs) {
	return lhs.re < rhs.re || lhs.re == rhs.re && lhs.im < rhs.im;
}
```

Теперь очень тривиально определяем оператор **>**:
```C++
bool operator>(const Complex& lhs, const Complex& rhs) {
	return rhs < lhs;
}
```

Таким образом с помощью одного оператора **<** можно определить все операторы сравнения. Оператор равенства определяется так:

```C++
bool operator==(const Complex& lhs, const Complex& rhs) {
	return !(lhs > rhs) && !((lhs < rhs));
}
```

Итого: общее правило для определения оператора сравнения это сначала определяется **<**, а с помощью него все остальное.

## Оператор SpaceShip
Позволяет автоматически определить операторы сравнения для класса.
Как мы заметили, все операторы сравнения определяются через один оператор **<**, так зачем же определять все остальные вручную, если они все одинаковые. Начиная со стандарта С++20 можно определить оператор **SpaceShip** (Thee-way comparison)
Для работы необходимо сделать `#include <compare>`

```C++
struct Complex {
	double re = 0.0;
	double im = 0.0;
	
	Complex(double re): re(re) {}
	Complex(double re, double im): re(re), im(im) {}
	
	Complex& operator+=(const Complex& other) {
		re += other.re;
		im += other.im;
		return *this;
	}
	
	// сигнатура SpaceShip
	std::weak_ordering operator<=>(const Complex& other) const = default;
	
	friend std::istream& operator>>(std::istream& in, Complex& copm);
};

Complex operator+(const Complex& lhs, const Complex& rhs) {
	Complex result = lhs;
	result += rhs;
	return result;
}  

std::ostream& operator<<(std::ostream& out, const Complex& comp) {
	out << comp.re << " + i*" << comp.im << '\n';
	return out;
}

std::istream& operator>>(std::istream& in, Complex& comp) {
	in >> comp.re >> comp.im;
	return in;
}
```

**SpaceShip** позволяет автоматически определить операторы сравнения лексикографически. Компилятор работает по следующему правилу: когда в коде появляется оператор сравнения, он сначала смотрит в определение конкретно этого оператора в классе, если нет, то ищет **SpaceShip**, в котором определен лексикографически этот оператор.
Лексикографическое сравнение выбрано из-за эффективности.

Возвращаемый тип у **SpaceShip** это 3 вида значений определяемых пользователем: **std::weak_ordering, std::strong_ordering, std::partial_ordering**. Это типы отношения порядка для элементов (далее математика..).
Но std::partial_ordering автоматически не определяется.

Отличие порядков сравнения в C++ и в математике немного отличается. В C++ нет несравнимых элементов. Тогда разница между std::weak_ordering и std::strong_ordering не очень понятна. А она есть: 
- std::weak_ordering отождествляет объекты, то есть по сути своей они равны, но имеют некоторые отличающиеся параметры (например std::string может иметь одинаковое строковое значение, но разный capasity, хотя строки равны). 
 - std::strong_ordering утверждает, что если объекты равны, то каждый их параметр совпадает

К сожалению SpaceShip сравнивает объекты по их полям (если переменные, то по значению, если указатели и ссылки, то по адресу) и не всегда получится обойтись default значением оператора (например для строк не подойдет default ибо поэлементно строки не сравниваются)

Допустим, что мы захотели определить оператор SpaceShip.
```C++
struct Complex {
	double re = 0.0;
	double im = 0.0;
	
	Complex(double re): re(re) {}
	Complex(double re, double im): re(re), im(im) {}
	
	std::weak_ordering operator<=>(const Complex& other) const {
		return std::weak_ordering::equivalent;
	}
};

int main() {
	// теперь хотим сравнить числа
	Complex a(2.0);
	Complex b(2.0);
	a == b; // ошибка!
}
```

А дело в том, что если SpaceShip определяется не defalt, то операторы == и != удаляются и требуют определения (остальные определяются дефолтно). Это связано с эффективностью: для разных типов нужно по-разному определять эффективные == и !=
