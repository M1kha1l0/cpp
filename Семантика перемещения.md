# Move-семантика
Идея такая: каждый раз вызывать оператор копирования при передаче, например, временного объекта очень дорого по памяти (двойное выделение памяти):

```cpp
#include <iostream>

class BigData {
	public:
	BigData(const BigData& other) {
		std::cout << "Create BigData\n";
	}
	BigData() {
		std::cout << "Create BigData\n";
	}
};

class A {
	BigData* d;
	public:
	A(): d(new BigData) {}
	A(const A& other): d(new BigData) {}
};

class B {
	A field;
	
	public:
	B(const A& other): field(other) {}
};

int main() {
	B b(A{});
}
```

Но по идее нам не нужно 2 раза выделять память, чтобы с помощью временного объекта проинициализировать новый.
Чтобы убрать данное недоразумение исправим копирование в классе A:

```cpp
#include <iostream>

class BigData {
	public:
	BigData(const BigData& other) {
		std::cout << "Create BigData\n";
	}
	BigData() {
		std::cout << "Create BigData\n";
	}
};

class A {
	BigData* d;
	public:
	A(): d(new BigData) {}
	A(A& other): d(other.d) { other.d = nullptr; }
};

class B {
	A field;
	
	public:
	B(A& other): field(other) {}
};

int main() {
	A a;
	B b(a);
}
```

Теперь все работает за одну операцию выделения памяти, но теперь мы не можем хранить 2 копии объекта и передавать временные объекты в качестве аргумента конструктора. Это беда.

Поэтому были придуманы **r-value** ссылки. По сути своей это флаг для компилятора, что мы хотим сделать перемещение ресурса, а не копирование.

```cpp
#include <iostream>

class BigData {
	public:
	BigData(const BigData& other) {
		std::cout << "Create BigData\n";
	}
	BigData() {
		std::cout << "Create BigData\n";
	}
};

class A {
	BigData* d;
	public:
	A(): d(new BigData) {}
	A(const A& other): d(other.d) { other.d = nullptr; }
};

class B {
	A field;
	
	public:
	B(A& other): field(other) {}
};

int main() {
	A a;
	B b(a);
}
```

На помощь приходит `std::move`, эта функция кастит l-value ссылки к r-value и тогда все работает:

```cpp
#include <iostream>

class BigData {
	public:
	BigData(const BigData& other) {
		std::cout << "Create BigData\n";
	}
	BigData() {
		std::cout << "Create BigData\n";
	}
};

class A {
	BigData* d;
	public:
	A(): d(new BigData) {}
	A(A& other): d(other.d) { other.d = nullptr; }
};

class B {
	A field;
	
	public:
	B(A& other): field(other) {}
};

int main() {
	A a;
	B b(a);
}
```

Реализация `std::move` **(реализация покрывает не все случаи, но точно большинство)**:

```cpp
template <typename T>
T&& move(T& obj) {
	return static_cast<T&&>(obj);
}
```

Сама по себе функция std::move позволяет перейти к реализации с функцией с move-семантикой.

std::move невероятно полезный инструмент, так например std::swap ускоряется с линейной сложности до константной:

```cpp
template <typename T>
void swap(const T& a, const T& b) {
	T tmp = std::move(a);
	a = std::move(b);
	b = std::move(tmp);
}
```

Вместо привычного копирования или передачи указателей, теперь просто перемещение ресурсов.

Далее в std::forward мы исправим функцию перемещения.

## std::forward

До std::forward существовала проблема.
Вспомним аллокаторы, метод construct (или emplace_back в векторе) принимает в себя указатель на объект и пакет аргументов. Зная только об std::move, наивно хочется перегрузить construct для T&& и T&, но число аргументов переменное и бессмысленно создавать $2^n$ перегрузок одной функции, чтобы дальше пробрасывать перемещение.
Итого требования: сохранить вид value переданного аргумента.

Для начала посмотрим на сигнатуру `construct`:

```cpp
template <typename T, typename... Args>
void construct(T* ptr, Args&&... args) {
	new (ptr) T(std::forward<Args>(args)...);
}
```

И вот эта корректная реализация этого метода. Здесь мы учитываем все переданные виды value в аргументы. Теперь мы сможем переместить ресурс там, где это необходимо с помощью `std::forward`. (Заметьте, что здесь есть [[(!)Типы выражений и ссылки|универсальная ссылка]])

Посмотрим теперь еще один пример применения `std::forward`. Допустим нужно хранить вектор строк и конструироваться из их произвольного количества без лишних копирований.

```cpp
#include <string>
#include <vector>

struct S {
	std::vector<std::string> v;
	
	template <typename... Args>
	S(Args&&... args) {
		v.reserve(sizeo...(args));
		// implement
	}
}
```

Понятно, что нужно placement выражение с push_back, которое бы обрабатывало и те строки, которые нужно переместить, а не копировать. Вот здесь то и нужен std::forward:

```cpp
#include <string>
#include <vector>

struct S {
	std::vector<std::string> v;
	
	template <typename... Args>
	S(Args&&... args) {
		static_assert(/*проверка типов с помощью std::conversable*/);
		v.reserve(sizeo...(args));
		(v.push_back(std::forward<Args>(args)), ...);
	}
}
```

Теперь пользуясь [[(!)Типы выражений и ссылки|правилами передачи универсальных ссылок]] реализуем `std::forward`:

```cpp
template <typename T>
T&& forward(T& value) {
	return static_cast<T&&>(value);
}
```

Исходя из правила коллапса ссылок мы обязаны принимать T& иначе просто невозможно будет вернуть T&&, т.к. он будет инициализироваться lvalue значением, что запрещено. Тогда ничего не остается, кроме как реализовать `std::move` и это действительно работает, т.к. правила коллапсирования ссылок для этого располагают.
**(Но и это не совсем верная реализация)**

Но поведение `std::forward` и `std::move` отличается. Первое дает нам искомый тип переданный в аргументы, а второй приводит все к rvalue ссылке. Поведение разное, а реализации одинаковые.

Исправим `std::move`. Сейчас, сели дать в move тип с одиночным амперсандом, то вернется lvalue ссылка, что противоречит поведению.

```cpp
template <typename T>
std::remove_reference_t<T>&& move(T&& value) {
	return static_cast<std::remove_reference_t<T>&&>(value);
}
```

Здесь нам придет на помощь метафункция для удаления ссылок с типа. Теперь мы всегда будем возвращать rvalue в независимости от того, что нам передали. Ну и чтобы передавали любой тип ссылок будем принимать универсальную ссылку, т.к. на выходе всегда получим rvalue ссылку.

Теперь исправим `std::forward`:

```cpp
T&& forward(std::remove_reference_t<T>& value) {
	return static_cast<T&&>(value);
}
```

Такая реализация нужна по нескольким причинам - защита от  неправильного использования. Если бы была просто ссылка, то мы бы всегда кастили к rvalue и перемещали, когда не надо.

Но у форварда есть еще одна перегрузка на случай если переданное значение уже rvalue:

```cpp
T&& forward(std::remove_reference_t<T>&& value) {
	static_assert(!std::is_lvalue_reference_v<T>);
	return static_cast<T&&>(value);
}
```

Это нужно, чтобы в forward не попадали rvalue значения переданные как lvalue.

## std::move_if_noexecpt

Сам по себе std::move не выбрасывает исключения, но конструкторы перемещения в классах вполне себе могут себе такое позволить.

В векторе, например, при реаллокации все данные перемещаются из одной памяти в другую, не копируются. Но как сохранить безопасность относительно исключений, когда объекты перемещаются? Переместить назад не вариант, исключения все еще могут выбрасываться и на обратном действии.

Для этого был специально придуман `std::move_if_noexecpt`, он перемещает объекты, если их конструктор перемещения работает без исключений, и копирует, если конструктор перемещения выбрасывает исключения.

Поэтому для объектов всегда нужно помечать, что он noexept, если он действительно работает без исключений.
