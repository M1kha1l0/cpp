# Recuires выражения
На замену enable_if пришли recuires выражения.
Рассмотрим на примере: мы хотим проверить, определена ли функция сложения над типом T:

```cpp
template <typename T>
requires requires(T a, T b) { a + b; }
T add(const T& a, const T& b) {
	return a + b;
}
```

Здесь requires это оператор, который принимает в себя какие-то аргументы и проверяет, что выражение в фигурных скобках компилируются. Здесь 2 оператора поскольку правый requires отвечает за выражение (оператор), которое проверяет, а левый принимает значение выражения (спецификатор, объявление сущности), которое после и определяет доступность метода. (По сути своей это логическое выражение)
В итоге он возвращает булевое значение результата проверки. Т.е. доступен такой вывод:

```cpp
#include <iostream>
#include <vector>

template <typename T>
requires requires(T a, T b) { a + b; }
T add(const T& a, const T& b) {
	return a + b;
}

template <typename T>
void test() {
	std::cout << requires(T a, T b) { a + b; };
}

int main() {
	test<int>(); // 1
	test<std::vector<int>>(); // 0
}
```

Кроме того, requires поддерживает обычные выражения, поскольку это оператор:

```cpp
template <typename T, size_t N>
requires (N <= 1'000'000)
struct Array {
	T arr[N];
};
```

Также этот оператор принимает значение мета-функции:

```cpp
template <typename T, size_t N>
requires std::is_same_v<T, long long>;
struct Array {
	T arr[N];
};
```

Но requires никогда не будет проверять корректность инстанцирования функций, его задача только проверить, определена ли функция с такими параметрами. Так, например, такой код будет компилироваться:

```cpp
#include <algorithm>
#include <list>

template <typename T>
requires requires(T a, T b) { std::sort(a, b); }
void my_sort(const T& a, const T& b) {}

int main() {
	std::list<int> l(10);
	
	my_sort(l.begin(), l.end());
}
```


Разберемся теперь, что вообще можно  писать внутри фигурных скобок. В них пишется любое выражение, которое компилятор будет проверять **только на компилируемость**, не на истинность. Код, приведенный ниже, будет компилироваться:

```cpp
template <typename T, size_t N>
requires requires {
	sizeof(T) == N;
}
void test(T a) {}

int main() {
	test<int, 10>(0);
}
```

Но можно писать вложенные requires и тогда проверка будет уже и на истинность:

```cpp
template <typename T, size_t N>
requires requires {
	requires sizeof(T) == N;
}
void test(T a) {}

int main() {
	test<int, 10>(0);
}
```

Также requires позволяет проверить типы переданные в шаблон:

```cpp
#include <vector>

template <typename T, size_t N>
requires requires {
	requires sizeof(T) == N;
	typename T::value_type;
}
void test() {}

int main() {
	test<std::vector<int>, 24>(); // OK
	test<int, 4>(); // CE
}
```

Теперь мы проверяем, что в T лежит using с именем value_type.

Кроме того, recuires может проверять возвращаемые типы выражений.
Важно, что когда мы проверяем возвращаемые типы метафункцией, то первый аргумент автоматически подставляется возвращаемый тип

```cpp
template <typename Func>

```

Итого мы имеем несколько случаев разного поведения requires:
-  Простой requires. Проверяется истинность выражения (одиночный оператор)
-  Requires со специализатором. Проверяется уже на компилируемость переданного выражения. Может содержать аргументы.
-  Вложенный requires. Объединенный в один предыдущие типы.
(примеры приведены раннее).

И что мы можем проверять:
-  Логические выражения
-  Выражения
-  Типы
-  Возвращаемые значения (в концептах)

Практический пример:
В функции необходимо создать unordered_set, это значит, что объект должен быть хешируемым с помощью std::hash, мы можем это проверить:

```cpp
template <typename T>
requires requires (T a) {
	std::hash<T>()(a);
}
void test() {}
```

Почему нельзя было просто написать `typename std::hash<T>`? Как обсуждали раннее, requires не инстанцирует шаблоны, а значит всегда можно создать std::hash от любого шаблона, этот requires бесполезен.

# Концепты
По сути своей это названные именем requires выражения, которые мы можем применять несколько раз. Кроме того, они служат логическим объединением свойств для шаблонного типа. Посмотрим на примере find_if:

```cpp
template <typename InputIt, typename Predicate>
InputIt find_if(InputIt begin, InputIt end, Predicate p) {
	for(auto it = begin; it != end;++it) {
		if(p(*it)) {
			return it;
		}
	}
	return end;
}
```

Здесь есть проблема. Вообще говоря название шаблонных параметров это всего лишь их логическое обозначение, что сюда стоит пихать, но совсем не ограничивающее. На место аргумента условия может встать, вполне себе, int.

Чтобы фильтровать, что нам не нужно, напишем концепт:

```cpp
template <typename T>
concept InputIterator = requires (T it) {
	++it;
	*it;
};

template <typename InputIt, typename Predicate>
requires InputIterator<InputIt>
&& requires (Predicate p, InputIt it) { p(*it); }
InputIt find_if(InputIt begin, InputIt end, Predicate p) {
	for(auto it = begin; it != end;++it) {
		if(p(*it)) {
			return it;
		}
	}
	return end;
}
```

Также можно проверять возвращаемые типы функций, с помощью -> (не оператор).
(Используются концепты из стандартной библиотеки)

```cpp
#include <concepts>

template <typename T>
concept InputIterator = requires (T it) {
	++it -> std::same_as<T&>;
	std::iterator_traits<T>::value_type;
	*it-> std::convertible_to<std::iterator_traits<T>::value_type>;
};

template <typename InputIt, typename Predicate>
requires InputIterator<InputIt>
&& requires (Predicate p, InputIt it) { p(*it); }
InputIt find_if(InputIt begin, InputIt end, Predicate p) {
	for(auto it = begin; it != end;++it) {
		if(p(*it)) {
			return it;
		}
	}
	return end;
}
```

Важно, что первый аргумент шаблонного концепта подставляется результатом выражения перед стрелкой.

Также концепты позволяют использовать сокращенную запись, чтобы не писать лишний раз их объявление. Это пишется на месте объявления шаблонного типа:

```cpp
#include <concepts>

template <typename T>
concept InputIterator = requires (T it) {
	{++it} -> std::same_as<T&>;
	std::iterator_traits<T>::value_type;
	{*it} -> std::convertible_to<typename std::iterator_traits<T>::value_type>;
};

template <InputIterator InputIt, typename Predicate>
requires (Predicate p, InputIt it) { p(*it); }
InputIt find_if(InputIt begin, InputIt end, Predicate p) {
	for(auto it = begin; it != end;++it) {
		if(p(*it)) {
			return it;
		}
	}
	return end;
}
```

Сократим теперь и requires на функцию (используем из библиотеки):

```cpp
#include <concepts>

template <typename T>
concept InputIterator = requires (T it) {
	{++it} -> std::same_as<T&>;
	std::iterator_traits<T>::value_type;
	{*it} -> std::convertible_to<typename std::iterator_traits<T>::value_type>;
};

template <InputIterator InputIt, std::predicate<decltype(*std::declval<It>())> Pred>
InputIt find_if(InputIt begin, InputIt end, Pred p) {
	for(auto it = begin; it != end;++it) {
		if(p(*it)) {
			return it;
		}
	}
	return end;
}
```

Но и это не предел, можно сократить с помощью шаблонной подстановки [[Вывод типов|auto]]:

```cpp
#include <concepts>

template <typename T>
concept InputIterator = requires (T it) {
	{++it} -> std::same_as<T&>;
	std::iterator_traits<T>::value_type;
	{*it} -> std::convertible_to<typename std::iterator_traits<T>::value_type>;
};

InputIt find_if(InputIterator auto begin, InputIterator auto end, std::predicate<decltype(*begin)> auto p) {
	for(auto it = begin; it != end;++it) {
		if(p(*it)) {
			return it;
		}
	}
	return end;
}
```

В данном случае концепт работает, т.к. поставлен перед авто и при этом сам объявляет тип, который передается. НО в данном случае 2 итератора не эквивалентны, т.к. авто независимые. Дополнительно нужно писать концепт на 2 одинаковых типа в аргументах, ну или использовать decltype, т.к. begin уже объявлен:

```cpp
#include <concepts>

template <typename T>
concept InputIterator = requires (T it) {
	{++it} -> std::same_as<T&>;
	std::iterator_traits<T>::value_type;
	{*it} -> std::convertible_to<typename std::iterator_traits<T>::value_type>;
};

InputIt find_if(InputIterator auto begin, decltype(begin) end, std::predicate<decltype(*begin)> auto p) {
	for(auto it = begin; it != end;++it) {
		if(p(*it)) {
			return it;
		}
	}
	return end;
}
```
