# lvalue и rvalue
Это категории выражений, а не типов. Тип сам по себе не обладает никаким value, только выражения их имеют. По сути lvalue это то, что лежит в памяти, а rvalue необязательно.

Формальное определение (для фундаментальных типов):
lvalue - название переменной и неважно какого типа (т.е. если выражение является переменой, то это lvalue). Также это операторы с присваиванием, префиксные операторы,  унарный оператор '*', оператор []. Также вызовы функций возвращаемый тип T& (для кастов то же самое)

rvalue - литералы (кроме строковых - они lvalue), все арифметические операторы и операторы сравнения над стандартными типами, а также постфиксные операторы, унарный амперсанд. Также вызовы функций у которых возвращаемый тип T или T&& (для кастов то же самое)

# R-value reference
Ссылка обладающая свойствами:
- Инициализируется только rvalue выражением.
- Инициализируется выражением из вызова функции.

Посмотрим на практике:

```cpp
int x = 5;
int&& y = x; // CE
int&& y = 6; // OK
y = 7;       // OK

int&& z = y; // CE - вид выражения lvalue
// Но можно так
int&& z = static_cast<int&&>(y);
```

Еще одно назначение типа `T&&` это продление жизни временного объекта.

Это ссылка, которая не имеет адреса в памяти и псевдонима. Как правило все временные объекты (результаты вычислений, результаты функций) являются r-value ссылками на объект. Получило свое название из-за того, что находится справа от выражения присваивания.

Такие ссылки обладают единственностью и имеют тип `T&&`.

```cpp
#include <iostream>

int main() {
	int&& rv = 10
	return 0;
}
```

Важный момент: при передаче в аргументы r-value ссылки она становится l-value.

```cpp
#include <iostream>

struct A {};

void g(A& x) {
	std::cout << "1\n";
}

void g(A&& x) {
	std::cout << "2\n";
}

void f(A&& x) {
	g(x);
}

int main() {
	f(A{});
}
```

Это связано с тем, что переданные аргументы в функцию сохраняются в области памяти (хоть и временной) соответственно имеют адрес в памяти, а значит r-value не являются.


# L-value reference
Ссылка. Ссылка является псевдонимом объекта. Обязательно должна быть инициализирована.

```cpp
#include <iostream>

int main() {
	int x = 10;
	int &y = x;
	std::cout << x << " " << y << '\n'; // "10 10"
	y = 25;
	std::cout << x << " " << y << '\n'; // "25 25"
	x = 23;
	std::cout << x << " " << y << '\n'; // "23 23"
	return 0;
}
```

UB:
```cpp
#include <iostream>

int& foo() {
	int i = 10;
	return i;
}

int main() {
	int& i = foo();
	std::cout << i; // undefinded behaivour
	return 0;
}
```

Поскольку ссылка определена на стеке, то после завершения `foo()` место освободится, а значит ссылка никуда не ведет и получается UB.

`const Type&` - делает **lvalue** ссылку доступной для конвертации в **rvalue** ссылку.
# Квалификаторы ссылок
Рассмотрим структуру, которая хранит в себе строку:

```cpp
#include <string>

struct S {
	std::string str;
	
	std::string get_data() const {
		return str;
	}
};
```

Мы хотим получить из структуры строку, которую она хранит. Например вызываемся так:

```cpp
std::string s = S{"abc"}.get_data();
```

При таком подходе вызывается слишком много копирований (конструктор, вывод функции и копирование в новую переменную). Хочется избежать лишнего копирования для временного объекта. Для этого есть квалификаторы ссылок текущего объекта, который позволяет вызывать методы объекта, когда он под определенной ссылкой:

```cpp
#include <string>

struct S {
	std::string str;
	
	std::string get_data() const {
		return str;
	}
	
	std::string get_data() && {
		return std::move(str);
	}
};
```

Теперь вторая перегрузка метода `get_data()` вызывается, если объект под rvalue ссылкой (скорее всего временный) и теперь, чтобы лишний раз не копировать, просто заберем ресурсы из временного объекта.

Есть и квалификатор и для lvalue:

```cpp
#include <string>

struct S {
	std::string str;
	
	// теперь метод вызывается от lvalue объекта 
	std::string get_data() & {
		return str;
	}
	
	std::string get_data() && {
		return std::move(str);
	}
};
```

Правило с константностью ссылок сохраняется:

```cpp
#include <string>

struct S {
	std::string str;
	
	// допустим вызов от rvalue и от lvalue
	std::string get_data() const & {
		return str;
	}
	
	std::string get_data() && {
		return std::move(str);
	}
};
```

Суть этих вызовов в том, что вы как будто принимаете свой объект по типу указанному в квалификаторе. И такая перегрузка корректная.
К тому же если есть перегрузка с ссылочным квалификатором, то квалификатор должен  быть у всех перегруженных методов.

Очень важное замечание на примере: Допустим определен класс A с перегруженным оператором +, тогда не будет ошибкой компиляции такое выражение:

```cpp
struct A {
	A operator+(const A& other) {
		return A{};
	}
};

int main() {
	A a, b, c;
	a + b = c;
}
```

А ведь это не является логичным, как можно rvalue присвоить значение. С точки  зрения языка все корректно, перегрузка оператора равенства не обладает ссылочным квалификатором, а значит принимает любое value.
Чтобы такое выражение не компилировалось добавим:

```cpp
struct A {
	A operator+(const A& other) {
		return A{};
	}
	
	A& operator=(const A& other) & {
		return *this;
	}
};

int main() {
	A a, b, c;
	a + b = c; // CE
	c = a + b; // OK
}
```

Таким образом мы запрещаем компилятору парсить корректно такие выражения. И получаем ошибку:

```bash
test.cpp: In function ‘int main()’:
test.cpp:34:17: error: passing ‘A’ as ‘this’ argument discards qualifiers [-fpermissive]
   34 |         a + b = c;
      |                 ^
test.cpp:27:12: note:   in call to ‘A& A::operator=(const A&) &’
   27 |         A& operator=(const A& other) & {
      |            ^~~~~~~~
```

# Универсальные ссылки
Для решения проблем с семантикой перемещения для пакетных шаблонов придуманы универсальные ссылки.

Из [[Семантика перемещения|файла]] будем рассматривать реализацию метода construct в [[(!)Аллокаторы|аллокаторах]]:

```cpp
template <typename T, typename... Args>
void construct(T* ptr, Args&&... args) {
	new (ptr) T(std::forward<Args>(args)...);
}
```

Заметим, что Args&& это не rvalue ссылка в чистом виде. Это является исключением из правил работы value ссылок.
Если в аргументы функции передан шаблонный параметр **ФУНКЦИИ** с двумя амперсандами, то это универсальная ссылка, которая может принимать в себя все виды value.

Когда ссылка не является универсальной:

```cpp
template <typename T>
class C {
	void f(T&&) {}
};
```

Здесь шаблонный аргумент T передан классу, а не функции и поэтому T&& принимает объекты только по rvalue ссылке.

Если добавляется константность или передаются другие объекты, которые принимают этот тип как шаблонный аргумент, то ссылка  тоже не является универсальной.
(т.е. в `f(const Args&&...)` нет универсальной ссылки и в `void f(std::vector<Args>&&)` тоже нет универсальной ссылки, даже если бы это был шаблонный параметр функции).

Рассмотрим на примере, какие типы мы получим при передаче по шаблонному пакету:

```cpp
#include <iostream>

template <typename T>
void h(T&& x);

template <typename T>
void g(T& x);

template <typename T>
void f(T x);

int main() {
	int x = 0;
	f(x); // T = int
	g(x); // T = int
	h(x); // T = int&
	
	int& y = x;
	f(y); // T = int
	g(y); // T = int
	h(y); // T = int&
	
	int&& z = std::move(x);
	f(z); // T = int
	g(z); // T = int
	h(z); // T = int&
	
	h(1); // T = int
	h(std::move(x)); // t = int
}
```

Заметим, что для подстановок в `h()` будет всегда `int&`. И даже в последнем случае, т.к. это все еще lvalue.

Это подводит нас к правилу reference collapsing:
```txt
T& + & -> &
T& + && -> &
T&& + & -> &
T&& + && -> &
```

Если на переданный тип шаблон навешивает сверху еще амперсандов, то они накладываются по такому правилу.

Но это правило не применяется, когда тип передан явно.

Кроме того, исходя из этого правила в универсальных ссылках компилятор дополнительно навешивает на  тип ссылку, чтобы в итоге получилась именно та, что передали. 

Эти правила наталкивают на мысль как правильно реализован [[Семантика перемещения|std::forward]].

Пример с перегрузкой по универсальной ссылке :

```cpp
#include <iostream>

template <typename T>
struct S {
	S() = default;
	
	S(const S&) {
		std::cout << 'A' << std::endl;
	}
	
	template <typename U>
	S(S<U>&&) {
		std::cout << 'B' << std::endl;
	}
	
	template <typename U>
	S(U&&) {
		std::cout << 'C' << std::endl;
	}
};

int main() {
	S<int> a;
	S<int> b(a);
}
```

Выведется "C". Не А, потому что подстановка лучше приведения типов (от не константной к константной), универсальная ссылка в С  может быть и не константной. Точно не В, поскольку ожидает rvalue, а передано lvalue.

# Дополнение до 5 категорий value
## gvalue
Generalized value. Для него lvalue является частным случаем и xvalue тоже. По сути своей это те виды value у которых под собой есть ячейка в памяти. Этим вкратце объясняется разница между lvalue и rvalue: у lvalue точно есть адрес в памяти, а у rvalue не обязательно.
## xvalue
Expired value. xvalue это частный случай gvalue и rvalue.
Формально можно перечислить от чего можно получить этот вид value. 
-  Результат вызова функции (если результат rvalue-ссылка).
-  Каст к rvalue-ссылке.
-  Обращение через оператор точки или стрелки или квадратных скобок от rvalue объекта.
По сути своей это rvalue с адресом в памяти, т.е. lvalue со свойствами rvalue.
Поэтому в примере 2 вызывается 2 конструктора, т.к. компилятор видит вызов от сконструированного объекта и ему нужно положить его в память.
## prvalue
Pure value. Является частным случаем rvalue.
Тот самый каноничный rvalue без имени и адреса в памяти.
##
Для понимания семантики перемещения это совершенно не нужно - rvalue и lvalue вполне достаточно.
Все эти категории необходимы для так называемого copy elision.
Рассмотрим на примере:

```cpp
// пример 1
// Сколько вызовов конструкторов?
std::string s = string("abc");
```

Исходя из предыдущих знаний понимаем, что вызовется 2 конструктора (конструктор от строки и конструктор перемещения).
Но на самом деле вызовется 1 конструктор от строки благодаря такому явлению как copy elision. В стандарте указано, что если объект при создании инициализируется prvalue выраением, то промежуточный объект создавать не обязательно, компилятор оптимизирует это присваивание.
Но если сделать так:

```cpp
// пример 2
std::string s = std::move(string("abc"));
```

То тип выражения примет xvalue и здесь действительно будет 2 конструктора, т.к. для передачи в мув нужен инициализированный объект.
Это и есть copy elision и с C++17 является обязательным.

То же самое происходит в возвращаемом типе функции:

```cpp
std::string f() {
	return std::string("abc");
}
```

Компилятор не будет вызывать дважды конструктор, чтобы инициализировать выходную строку. Если же тут тоже поставить std::move, то будет снова xvalue.

Теперь выражения вида:

```cpp
struct S {};

int main() {
	S s = S(S(S(S(S(S(S()))))));
}
```

Вызывает только 1 конструктор.

Но как компилятор понимает, когда нужно оптимизировать copy elision, а когда нет?
Это можно представлять как вычисление в уме: высчитаете определенное выражение полностью в уме и конечный результат записываете на бумажку, чтобы запомнить. Оптимизация компилятора работает логически также, но с нюансом. Для этого реализован механизм temporary materialization - это неявное приведение типов выражения из xvalue в prvalue и наоборот. 
Этот механизм вызывается только в определенных сценариях: если prvalue такого же типа как и объект, то оно сконвертируется к xvalue объекту
-  Если инициализируется сслыка через prvalue.
-  Если через prvalue обращаемся через точку.
-  Если результат типа prvalue игнорируется.
-  ...
Таких сценариев много, предлагается посмотреть их на [cppreference.com](https://cppreference.com/)

Но если инициализируется из prvalue, то temporary materialization нет.

## lvalue to rvalue
Также возможна конвертация lvalue к rvalue. Как это понимать:
По сути своей это чтение из памяти, у вас есть какие-то данные в памяти и надо их перенести в регистры процессора (концептуально, не является действительным).
Это не работает, когда передается lvalue в функцию, которая принимает lvalue. Здесь это можно сделать только явно.

## RVO (return value optimization)
Это частный случай copy elision. Мы рассматривали его на примере 2 выше. Стандартом это не гарантируется, но многие компиляторы так делают.
Из-за этой оптимизации не стоит без причины писать в конце std::move, RVO в таком случае не сработает.

Кроме того оптимизируются и именованные переменные:

```cpp
std::string f() {
	std::string str = "abc";
	return str;
}
```

Когда функция возвращает по значению локальную переменную, то происходит оптимизация. Здесь тоже не надо писать std::move иначе оптимизация не работает.
Но RVO будет, если возвращаемый тип однозначен. И даже так будет оптимизация, если не удается сделать предыдущую оптимизацию, то возвращаемое значение инициализируется конструктором перемещения.

Но std::move нужен если принимаемый аргумент это rvalue ссылка:

```cpp
struct S{};

S f(S&& a) {
	// значение копируется
	return a;
	// а тут перемещаем
	return std::move(a);
}
```

Это происходит, потому что компилятор не считает переменную переданную по rvalue-ссылке локальной, а значит оптимизации нет.
