# dynamic_cast

Нужен тогда, когда под ссылкой на родительский объект гарантированно лежит объект наследника. Такой каст позволяет сделать доступным обращение из скастенного объекта к полям наследника.

Пример корректного применения:

```cpp
#include <iostream>

struct Base {
	int x = 0;
	
	virtual void f() {}
	virtual ~Base() = default;
};

struct Derived : Base {
	int y = 0;
	
	void f() override {}
};

int main() {
	Derived d;
	Base& b = d;
	dynamic_cast<Derived&>(b);
}
```

В данном случае каст корректный, поскольку за ссылкой на Base действительно стоит Derived. Теперь возможно обращение к полю **y** из Derived, т.е. корректно обрабатывается `int y = dynamic_cast<Derived&>(b).y;` и получим действительно **y** из Derived.

Важно, что **dynamic_cast** кидает исключение (std::bad_cast) в случае, если под ссылкой на Base лежит не Derived, а что-то другое (например Base), в отличии от static_cast, который вызовет UB и в рантайме это никак не проверить. 
Как и любой каст dynamic_cast работает и для не полиморфных объектов в случае каста к родителю от наследника (как и любой другой), но в таком случае использовать его не логично, ибо static_cast-та будет достаточно. Но стоит учитывать обратную ситуацию.

Чтобы проверить какой объект лежит под ссылкой на базовый класс можно воспользоваться кастом к указателю на Derived. dynamic_cast в случае, если внутри лежит не Derived вернет nullptr.

Применим dynamic_cast только к полиморфным объектам (только у которых есть виртуальные функции). Это логично просто исходя из того, что виртуальность в рантайме позволяет определить какой этот объект на самом деле (за счет виртуальных таблиц). Если же базовый класс не имеет виртуальных методов, то увы dynamic_cast выдаст CE (source type is not polymorhic). 
И из-за этого dynamic_cast очень дорогая операция. Ей необходимо в рантайме сходить по указателю на виртуальную таблицу, по ней побродить и понять, что это за объект, и уже после этого передвинуть указатель.
**Уточнение**: обязан быть полиморфным только объект от которого кастят.

Очень полезное свойство dynamic_cast это каст "вбок". Допустим есть ромбовидное наследование: бабушка -> мама, папа -> сын. И внезапно нужно скастить ссылку на маму к папе. static_cast-ту неоткуда знать, что мама и папа на самом деле совместимые типы, а dynamic_cast пройдет к объекту сына и т.к. он знает все сдвиги в памяти еще на этапе компиляции, то может от сына перейти к папе.
И тут есть примеры, когда 3 каста дают разные результаты:
Каст "вбок": dynamic_cast отрабатывает корректно, static_cast выдает CE, reinterpret_cast, как всегда, UB.
Каст вниз от виртуального предка: dynamic_cast отрабатывает корректно, static_cast выдает CE, reinterpret_cast, как всегда, UB.

Также любой полиморфный объект можно кастить к void*, при том только полиморфный. Таким образом можно проверять объект на полиморфность.

Как устроен dynamic_cast понятно из разбора [[ABI]].
 
Заметим, что dynamic_cast и полиморфизм очень тесно связаны и на этом построен целый механизм RTTI (RunTime Type Information). Этот механизм был специально создан, чтобы пользователь не следил за типом объекта вручную и не заводил дополнительных полей в объектах.

С помощью этого механизма нам доступно теперь спрашивать какой тип лежит под ссылкой без каста. Это оператор `typeid`

```cpp
#include <iostream>

struct Base {
	int x = 0;
	
	virtual void f() {}
	virtual ~Base() = default;
};

struct Derived : Base {
	int y = 0;
	
	void f() override {}
};

int main() {
	Derived d;
	Base& b = d;
	
	std::cout << typeid(b).name() << '\n'; // 7Derived
	// число 7 перед Derived это длина имени
}
```

Этот оператор возвращает специальный тип `std::type_info`, который из полезного содержит только оператор сравнение и метод name.

В отличии от dynamic_cast, typeid работает и с не полиморфными объектами. В таком случае оператор просто выведет название типа переменной, но преобразованное под устройство компилятора.

Так, например, `typeid(int).name()` выведет  "i".
