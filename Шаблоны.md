Идея шаблонов: нужно писать код, который одинаково работает для произвольных типов (мета-переменных типа). Это парадигма программирования.

# Объявление
Для шаблонов некоторая переменная **T** (тип) является именем мета-переменной. Нужно понимать, что для вызовов для разных типов генерируется разный ассемблерный код.

Шаблонными могут быть и функции и классы.
Пример использования шаблона функции:

```cpp
// не очень эффективная реализация
// из-за возможного нетривиального копирования
// модифицируем позже
template<typename T>
void swap(T& x, T&y) {
	T t = x;
	x = y;
	y = t;
}

template<typename T>
T max(T& x, T&y) {
	return x > y ? x : y;
}
```

Здесь мы говорим следующее: "Для любых произвольных типов **T** определена функция **swap** и **max**". 

Пример шаблонных классов (простейший это вектор):

```cpp
template <typename T>
class vector {
	T* arr;
	size_t sz;
	size_t cap;
};
```

Важно понимать, что при использовании шаблона для класса, генерируются разные классы, которые друг с другом не связаны и поэтому нельзя, например, приравнивать классы с разными шаблонами (но опять же можно это определить явно, автоматически не получится).

 В C++11 можно  делать шаблоны using-ов:
 
```cpp
template <typename T>
using mymap = std::map<T, T, T>
```

Кроме того, можно объявлять несколько мета-переменных типа через запятую:

```cpp
template <typename T, typename U>
void func(T x, U y) {
	// some code
}
```

# Использование
Для использования шаблонных функций не требуется явное указание типа, компилятор сам подставит нужный тип в функцию, т.к. это доступно на этапе компиляции.
Но если шаблон содержал только один тип, а подали 2 разных, то это ошибка компиляции, неявных преобразований в таком случае не производится, т.к. ищется шаблон.

```cpp
template<typename T>
void swap(T& x, T&y) {
	T t = x;
	x = y;
	y = t;
}

int main() {
	int x;
	int y;
	long z;
	
	swap(x, y); // OK
	swap(x, z); // CE
}
```

Но, если нужно, можно явно указать какой шаблон использовать:
```cpp
template<typename T>
void swap(T& x, T&y) {
	T t = x;
	x = y;
	y = t;
}

int main() {
	int x = 10;
	int y = 20;
	long z = 12l;
	
	swap(x, y); // OK
	swap<long>(x, z); // OK
}
```

И теперь ошибки для вызова нет, но это уже не касается шаблонов (проблема в конвертации типов ссылки, если бы передавали значение, то ошибки бы не было). Здесь явное указание шаблона и только теперь может быть неявное преобразование типов переменных, т.к. код уже сгенерирован.

Также доступно указание типа шаблона по умолчанию:

```cpp
#include <iostream>

template <typename T, typename U = int>
U f(T x) {
	std::cout << '1';
}

int main() {
	int x = 0;
	f<int>(x);
}
```

Для шаблонов с несколькими типами такие же правила передачи аргументов, как и для функций.
Но есть пример, где можно указывать не все типы:

```cpp
#include <iostream>

template <typename U, typename T>
U f(T x) {
	std::cout << '1';
}

int main() {
	int x = 0;
	f<int>(x);
}
```

Здесь достаточно задать явно только один тип, т.к. тип переменной **x** компилятор в состоянии определить самостоятельно. Но вот определить возвращаемое значение автоматически он не может.
# Что под капотом
По сути своей шаблоны это кодогенерация. На стадии компиляции, сразу после препроцессинга, компилятор генерирует код для вызванных шаблонов функций и по факту мы получаем разные функции с разными адресами в памяти и разный код. Причем код генерируется на основе вызова функции.
Сначала вызывается перегрузка шаблонов, потом инстанцирование, и потом уже специализация. 

# Перегрузка шаблонных функций

```cpp
#include <iostream>

template <typename T>
void f(T x) {
	std::cout << '1';
}

void f(int x) {
	std::cout << '2';
}

int main() {
	int x = 0;
	f(x);
}
```

Здесь нет ошибки компиляции. В данном случае выведет **2**. И это логично, общий случай работает для всех, но частный случай должен быть в приоритете.

```cpp
#include <iostream>

template <typename T>
void f(T x) {
	std::cout << '1';
}

void f(long long x) {
	std::cout << '2';
}

int main() {
	int x = 0;
	f(x);
}
```

Здесь наоборот выведется **1**, т.к. компилятор будет генерировать код для идеального соответствия типов (избегает конверсий).

Но если явно указывать, что генерировать надо с конкретным типом, то это обязывает компилятор сгенерировать код ибо выбора уже нет и это не ошибка компиляции.

```cpp
#include <iostream>

template <typename T>
void f(T x) {
	std::cout << '1';
}

void f(int x) {
	std::cout << '2';
}

int main() {
	int x = 0;
	f<int>(x); // 1
}
```


**ВАЖНО!**
При перегрузке шаблонных функций выбирается более частная.
Например:

```cpp
#include <iostream>

template <typename T, typename U>
void f(T x, U y) {
	std::cout << '1';
}

template <typename T>
void f(T x, T y) {
	std::cout << '2';
}

int main() {
	int x = 0;
	f(x, x); // 2
}
```
# Специализация шаблонов
Это нужно, когда требуется реализация отличная от общего представления. Например `vector<bool>`, где каждое значение в массиве меняет бит в памяти.
Это делается так:

```cpp
template <typename T>
class vector {
	T* arr;
	size_t sz;
	size_t cap;
};

// указываем, что не нужно дополнительно определять тип
template <>
// указываем явно какие аргументы принимает класс
class vector<bool> {
	char* arr;
	size_t sz;
	size_t cap;
};
```

Специализация также может быть частичной:

```cpp
template <typename T, typename U>
struct S {};

// Частичная специализация
// В скобках указываем, что все еще надо определить
template <typename W>
struct S<int, W> {};

template <typename W>
struct S<W, W> {};
```

Также частичная специализация может быть по ссылкам:

```cpp
template <typename T>
struct S {};

template <typename T>
struct S<T&> {};
```

В этом случае будут сгенерированы разные реализации для объектов с ссылочным типом и обычной переменной.
Но опять же, специализации выбираются по правилу чтения полного типа переменных.

Частичная специализация доступна только классам, для функций существует перегрузка и частичная специализация лишена смысла.

Для специализаций важен порядок, например:

```cpp
#include <iostream>

template <typename T, typename U>
void f(T, U) {
	std::cout << 1;
}

template <>
void f(int, int) {
	std::cout << 3;
}

template <typename T>
void f(T, T) {
	std::cout << 2;
}

int main() {
	f(0, 0); // 2
}
```

Здесь выведется **2**, поскольку специализация **(int, int)** указана для первого шаблона (в такой расстановке) и компилятор выберет шаблон с одинаковым типом, т.к. он наиболее подходящий. А в таком случае:

```cpp
#include <iostream>

template <typename T, typename U>
void f(T, U) {
	std::cout << 1;
}

template <typename T>
void f(T, T) {
	std::cout << 2;
}

template <>
void f(int, int) {
	std::cout << 3;
}

int main() {
	f(0, 0); // 3
}
```

Вывод будет **3**, т.к. это специализация шаблона с одинаковыми типами, который является специализацией шаблона для разных типов. Т.е. правило такое: выбирается наиболее частная подходящая специализация для вызова.

ОБЩЕЕ ПРАВИЛО ВЫБОРА ШАБЛОНА:
1) Сначала перегрузка между шаблонами (функции и шаблон).
2) Потом в выбранный шаблон подставляются типы и генерируются нужные версии.
3) Потом перегрузка из того, что получилось.

# Не типовые параметры шаблонов
Не только типы данных могут быть параметром шаблона. 
## Числовые
Так например в `std::array<int, 100>` создается массив из 100 целых чисел.
Как это написано:

```cpp
template <typename T, size_t N>
class array {
	T arr[N];
};
```

И это создаст на стеке массив с нужным размером.

Другой пример - матрицы:

```cpp
template <size_t M, size_t N, typename Field = double>
class Matrix {
	Field data[M][N];
};

template <size_t N, typename Field = double>
using SquareMatrix = Matrix<N, N, Field>;

int main() {
	// два разных класса
	Matrix<5, 5> m;
	SquareMatrix<5> sm;
}
```

Из такого объявления мы получили 2 различных класса, которые не связаны друг с другом и таким образом запретили конверсию между ними.
Но как теперь сделать умножение, ведь можно умножать квадратные матрицы на прямоугольные?

```cpp
template <size_t M, size_t K, size_t N, typename Field>
Matrix<M, N, Field> operator*(const Matrix<M, K, Field>& a, const Matrix<K, N, Field>& b);
```

Таким образом сразу же решается проблема с проверкой размерности матриц, это сразу будет CE, т.к. не будет существовать такого шаблона.
Но для шаблонных числовых параметров должны подаваться только константные значения, которые известны на этапе компиляции.

ВАЖНО !
Параметры шаблона принимают только целочисленные (интегральные) аргументы.
Но с C++20 можно и литералы тоже.

## Шаблонные параметры
Разберем на примере реализации stack
Допустим мы хотим иметь в качестве параметра контейнер, поверх которого будет оборачиваться стек (стек может быть реализован, как на деке, так и на списке).

Чтобы задать тип контейнера нужно написать так:

```cpp
template <typename T, template<typename> typename Container>
class Stack {
	Container<T> container;
};
```

Здесь вторым аргументом указано, что параметром стека является другой тип, который тоже имеет шаблонные параметры.
Его можно сделать по умолчанию:

```cpp
template <typename T, template<typename, typename> typename Container = std::vector>
class Stack {
	Container<T, std::allocator<T>> container;
};
```

Тип в векторе передавать не нужно, т.к. это будет объект, а нужно передать шаблон. `std::alllocator` это часть STL контейнера и пока знать о ней не нужно, но это является частью шаблона и при передаче шаблона нужно учитывать и аргументы по умолчанию.

Лучше написать так:
```cpp
template <typename T, template<typename> typename Container = std::vector>
class Stack {
	Container<T> container;
};

template <typename T>
using myvector = std::vector<T>;

int main() {
	// теперь компилируется и лучше работает с другими контейнерами
	Stack<int, myvector> s;
}
```

# Вычисления на шаблонах
Разберем на примере Фибоначчи

```cpp
template <int N>
struct Fibonacci {
	static const int value = Fibonacci<N-1>::value + Fibonacci<N-2>::value;
};
```

У шаблонов тоже есть рекурсия и она фиксированной глубины. И такой код не скомпилируется - будет вызвана ошибка глубины рекурсии шаблона. Значит требуется добавить специализацию для конечной рекурсии:

```cpp
template <int N>
struct Fibonacci {
	static const int value = Fibonacci<N-1>::value + Fibonacci<N-2>::value;
};

template <>
struct Fibonacci<0> {
	static const int value = 0;
};

template <>
struct Fibonacci<1> {
	static const int value = 1;
};
```

(Переменные все static т.к. объекты не создаются, а значение нужно сделать частью типа).
Такое вычисление работает за линейное время и генерирует 21 класс чисел Фибоначчи, что по сути своей мало отличимо от динамики.

# Зависимые имена
Рассмотрим такой пример:

```cpp
template <typename T>
struct S {
	using A = int;
};

template <typename T>
void f() {
	S<T>::A* x; // declaration or expression?
}

int main() {
	f<int>(); // CE
}
```

В данном выражении компилятор еще на стадии парсинга кода выдаст ошибку, потому что не понятно, что именно здесь требуется. Под **А** может лежать не только тип, а какая-нибудь переменная и тогда это будет уже выражение, а не ожидаемое объявление.

Но мы же хотим тип, а не выражение. Тогда нужно написать такую конструкцию:

```cpp
template <typename T>
struct S {
	using A = int;
};

template <typename T>
void f() {
	typename S<T>::A* x; // declaration
}

int main() {
	f<int>(); // OK
}
```

Перед зависимым именем нужно ставить typename, чтобы компилятор парсил это выражение как объявление. Без этого по умолчанию это парсится как выражение, потому что должна быть однозначность до подстановки типа, иначе в разных подстановках это может быть как выражение и как объявление, что не хорошо.

Если же под юзингом лежит переменная, то вызовется CE с указанием на объявление, которое внезапно стало выражением:

```cpp
template <typename T>
struct S {
	using A = int;
};

template <>
struct S<double> {
	static const int A = 5;
};

template <typename T>
void f() {
	typename S<T>::A* x; // declaration
}

int main() {
	f<int>(); // OK
	f<double>(); // CE
}
```

Еще один пример с парсингом в выражение, а не тип:
```cpp
#include <array>

template <typename T>
struct S {
	template<int N>
	using A = std::array<int, N>;
};

template <typename T>
void f() {
	S<T>::A<10> x; // declaration
}

int main() {
	f<int>();
}
```

А вот тут объявление становится выражением: `A < 10 > x`. И даже typename здесь не поможет, потому что компилятор еще не знает о названиях шаблонов. И для этого нужно написать template:

```cpp
#include <array>

template <typename T>
struct S {
	template<int N>
	using A = std::array<int, N>;
};

template <typename T>
void f() {
	S<T>::template A<10> x; // declaration
}

int main() {
	f<int>(); // OK
}
```

Еще пример с шаблонной функцией:

```cpp
template <typename T>
struct S {
	template <int N>
	void foo(int) {}
};

template <typename T>
void bar(int x, int y) {
	S<T> s;
	s.foo<5>(x+y);
};
```

И здесь тоже без явного указания, что происходит будет парситься так: `s.foo < 5 >(x+y)` - логическое выражение.
Решается это так:

```cpp
template <typename T>
struct S {
	template <int N>
	void foo(int) {}
};

template <typename T>
void bar(int x, int y) {
	S<T> s;
	s.template foo<5>(x+y);
};
```

Общая проблема такая: на этапе синтаксической проверки компилятору нужно отличать объявление от выражения и в случае подстановки типов через юзинги или вызов шаблонных функций это не всегда понятно. Компилятору на этапе синтаксической проверки ничего не известно о типах и структуре кода.

Эта проблема была решена в некоторых случаях начиная с C++11.

Также есть проблема с наследованием от шаблонного родителя:

```cpp
template <typename T>
struct Base {
	int x = 0;
};

template <typename T>
struct Derived : Base<T> {
	void f() {
		++x;
	}
};
```

В данном случае компилятор при проходе не поймет, что это вообще за **x** такой. Ведь его значение и смысл зависит от специализации и вообще его может не быть. В таких случаях исправлют это так:

```cpp
template <typename T>
struct Base {
	int x = 0;
};

template <typename T>
struct Derived : Base<T> {
	void f() {
		++this->x; // или
		++Base<T>::x;
	}
};
```

**Итого**, компиляция шаблонного кода происходит в 2 пробежки: проверка синтаксиса и проверка после подстановки шаблонных аргументов.

# Метафункции и type traits

Метафункции - функции от типов. Отличаются от простых функций тем, что принимают на вход типы и возвращают типы или значения переданные под типами, **не объекты**.

Например: простейший пример это проверка типа переменной

```cpp
template <typename T, typename U>
struct is_same {
	static constexpr bool value = false;
};

// специализация для одинаковых типов
template <typename T>
struct is_same<T, T> {
	static constexpr bool value = true;
};
```

Это реализация функции `std::is_same`, которая позволяет во время компиляции проверить типы на идентичность. Механизм основан на шаблонах, для одинаковых типов мы инстанцируемся в `is_same<T,T>` и выведем **true**.
Можно было бы, конечно, использовать `typeid`, но типы нам известны на этапе компиляции и тогда незачем ловить их равенство в рантайме. 

Применение следующее:
```cpp
template <typename T, typename U>
void f(const T& x, const U& y) {
	// ...
	if constexpr (is_same<T, U>::value) {
		x = y;
	}
	// ...
}
```

Здесь `constexpr` играет ключевую роль. Если типы не равны, то компилятор не будет компилировать код внутри if в этой инстанции, а значит внутри не будет ошибок компиляции. Если же убрать constexpr, то код не будет компилироваться при переданных типах, например, int и std::string.
Важно, чтобы условие для constexpr определялось на этапе компиляции.

Рассмотрим еще одну функцию `remove_reference`

```cpp
template <typename T>
struct remove_reference {
	using type = T;
};

template <typename T>
struct remove_reference<T&> {
	using type = T;
};
```

Данная функция позволяет удалить упоминание о ссылке у передаваемого типа. Пример использования:

```cpp
template <typename T>
void f() {
	typename remvove_reference<T>::type x;
}
``` 

В случае этой метафункции, ее возвращаемое значение это тип и это зависимое имя, а значит нужно в начале написать typename. Переданное значение в шаблон будет определяться как тип обычной переменной, а не ссылки.

Также есть еще одна полезная метафункция `remove_const`:

```cpp
template <typename T>
struct remove_const {
	using type = T;
};

template <typename T>
struct remove_const<const T> {
	using type = T;
};
```

Ничего схожего с const_cast тут нет.

Очень полезная функция `std::conditional`:

```cpp
template <bool B, typename T, typename F>
struct conditional {
	using type = F;
};

template <typename T, typename F>
struct conditional<true> {
	using type = T;
};
```

Но теперь посмотрим на еще одно улучшение для таких метафункций. Каждый раз писать перед вызовом typename не очень то и удобно, поэтому для каждой метафункции был написан отдельный юзинг, который решает эту проблему:

```cpp
template <bool B, typename T, typename F>
using conditional_t = typename conditional<B, T, F>::type;
```

И для каждой такой есть. Поэтому метафункции в чистом виде использовать не очень хорошо, но можно.
Также для метафункций с возвращаемым типом bool тоже написаны такие юзинги:

```cpp
template <typename T, typename U>
const bool is_same_v = is_same<T, U>::value;
```

Это все дело называется type trait.

# Variadic templates
Это шаблоны с переменным количеством аргументов.

```cpp
template <typename... Types>
void f(Types... tx) {}
```

Здесь **Types** это особый вид шаблонного аргумента, который представляет из себя пакет типов переданных в шаблон. Распаковка происходит так, как указано в аргументах функции (`Types...` это тип данных пакета).
Также можно этот пакет передать дальше по сути распаковав его в другую функцию:

```cpp
template <typename... Types>
void f(Types... tx) {
	g(tx...);
}
```

Рассмотрим более реальный пример: аналог printf на шаблонах

```cpp
#include <iostream>

template <typename Head, typename... Tail>
void print(const Head& head, const Tail&... tail) {
	std::cout << head << ' ';
	print(tail...);
}
```

Здесь мы будем перебирать весь пакет рекурсивно, так как мы сначала всегда печатаем первый переданный элемент, то обозначим его за какой-то отдельный тип, а оставшийся пакет пробросим дальше.
Это необходимо за тем, что 10 переданных аргументов в шаблоне отличаются от 9 и т.д.

Но есть еще важный нюанс: так как это рекурсия, то у нас должен быть базовый случай, т.к. переданный пакет может быть пустым и тогда Head уже не будет и это ошибка компиляции. Напишем его:

```cpp
#include <iostream>

void print() {}

template <typename Head, typename... Tail>
void print(const Head& head, const Tail&... tail) {
	std::cout << head << ' ';
	print(tail...);
}
```

Разберем еще  один пример на вариативные шаблоны. Напишем метафункцию, которая возвращает true, если все переданные типы одинаковые:

```cpp
template <typename First, typename Second, typename... Args>
struct is_homogeneous {
	static const bool value = std::is_same_v<First, Second>
							&& is_homogeneous<Second, Args...>::value;
};

template <typename First, typename Second>
struct is_homogeneous<Fisrt, Second> {
	static const bool value = std::is_same_v<First, Second>;
};
```

Здесь же базовый случай рекурсии определяется специализацией структуры для 2 типов.

Работает это все дело за линейное время, но обрывается, если на какой-то стадии выходит false. Но это не значит, что не все шаблоны не инстанцируются. Это можно исправить с ухищрениями.

Также для пакетов существует оператор `sizeof...()`, который выводит размер пакета.
## Выражения свертки

Считается, что предыдущие реализации устаревшие. Поэтому перепишем их на новый стандарт C++17.

Но сначала разберем на более простом примере. Проверим, что все аргументы указатели:

```cpp
#include <type_traits>

template <typename... Types>
struct all_pointers {
	static const bool value = (std::is_pointer_v<Types> && ...);
}
```

Теперь мы можем заметить, что в выражение добавилось математическое "и так далее". По сути своей это выражение подставит все остальные типы за бинарным оператором (бинарный оператор появится между ними тоже). Это называется fold expression.

Теперь разберем же наш is_homogeneous:

```cpp
template <typename Head, typename... Tail>
struct is_homogeneous {
	static const bool value = (std::is_same_v<Head, Tail> && ...);
};
```

Вышло гораздо короче, но по сути своей выполняет одно и то же, но не явно.

Теперь прокачаем наш print с выражением свертки:

```cpp
#include <iostream>

template <typename... Types>
void print(const Types&... type) {
	(std::cout << ... << type);
}
```

Выражение свертки может быть различным для разных операторов в зависимости от их ассоциативности (Подробнее [здесь](https://en.cppreference.com/w/cpp/language/fold.html) ).

# Паттерн CRTP

Это отдельный вид полиморфизма, называемый статическим полиморфизмом, поскольку в данном паттерне не происходит поиска метода по виртуальной таблице в рантайме, какая функция будет вызываться определяется на этапе компиляции.

Посмотрим как это устроено:

```cpp
template <class T>
struct Base {
	void inteface() {
		static_cast<T*>(this)->implementation();
	}
	
	static void static_func() {
		T::static_sub_func();
	}
};

struct Derived : Base<Derived> {
	void implementation();
	static void static_sub_func();
};
```

В данной реализации полиморфного наследования мы просто просим компилятор, чтобы он подставил на место T наш класс наследник Derived и во всех этих случаях при обращении к Base от Derived будет вызываться метод Derived, хоть и через interface или другие методы Base.

Может показаться, что эта штука вообще не должна компилироваться, но нет, как сказано выше это просто просьба подставить на T Derived. А вот если создавать объект типа Derived в поле Base, то будет уже ошибка компиляции ибо такую рекурсивную ситуацию разрешить невозможно. Но всегда можно объявить поле указателя на Derived или ссылку на Derived, определение Derived для этого не требуется.

Применяется для тех случаев, когда нужно создать один общий интерфейс для работы с полиморфным классом.

Еще один пример на использование CRTP: на википедии про вектор на шаблонах.
