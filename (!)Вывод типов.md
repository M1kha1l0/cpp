# auto
В C++ есть ключевое слово **auto**, которое принимает тип, который передали переменной с помощью какого-от выражения или возврата от функции. Естественно сам тип под auto определяется на этапе компиляции.

Разберемся как работает:

```cpp
int main() {
	auto x = 5;
}
```

Это достаточно просто. Этот механизм уже есть в языке - шаблоны. Компилятор автоматически понимает, что ему поставить. Эта запись работает также, как и:
`template<typename T> void f(T x) {}`

Интереснее ситуация с ссылочными типами:

```cpp
int main() {
	auto x = 5;
	auto y = x;        // int
	auto& z = x;       // int&
	const auto& a = x; // const int&
}
```

Так как это работает как шаблонная подстановка, то при подстановке типа отбрасывается ссылка у типа.

Самое интересное:

```cpp
int main() {
	auto x = 5;
	auto&& y = x;
}
```

Ошибки компиляции не будет. Здесь, собственно как и в шаблонах, `auto&&` это универсальная ссылка и правила для нее те же. Если бы дали `auto&& = std::move(x)`, то будет int&&.

Начиная с C++14 возвращаемый тип функции может быть auto:

```cpp
template <typename T>
auto f(T x) {
	return ++x;
}

template <typename T>
auto& g(T x) {
	return ++x;
}

int main() {
	auto x = f(1); // 2, вернется инт
	auto& y = f(1); // CE, нельзя lvalue инициализировать rvalue
	auto& y = g(1); // UB, битая ссылка на временный x
	auto&& z = g(1); // UB, битая ссылка
}
```

Auto не может возвращать сразу 2 типа из функции:

```cpp
template <typename T>
auto f(T x) {
	if(x < 0) {
		return 0;
	}
	else {
		return 1u;
	}
}

template <typename T>
auto g(int x) {
	if(x < 0) {
		return 0;
	}
	else {
		return 1u;
	}
}
```

Функция f выведет ошибку компиляции только при вызове, а g не даст скомпилироваться никак, неоднозначно определяется возвращаемое значение: int или unsigned int. Это не работает  даже при приводимых типов.

Но 2 возвращаемых значения все же можно сделать в шаблонном варианте кода:

```cpp
template <typename T>
auto f(T x) {
	if constexpr (std::is_same_v<T, int>) {
		return 0;
	}
	else {
		return 1u;
	}
}
```

Но есть нюанс, это будут 2 разные функции.

Почему это вообще работает. Auto для того чтобы определить какой возвращаемый тип, должен  просмотреть весь код функции и найти return, чтобы посмотреть тип значения. В это же время происходит компиляция `if constexpr`, а значит возвращаемый тип определяется на этапе компиляции и все работает.

НО. если мы попытаемся написать рекурсивную функцию с auto, то это не сработает. Компилятор защищен от рекурсированного определения типа auto и выведет ошибку.

```cpp
// не скомпилируется
template <typename T>
auto f(T x) {
	if constexpr (std::is_same_v<T, int>) {
		return f(x);
	}
	else {
		return 1u;
	}
}
```

В C++20 появилась возможность писать в аргументах функций auto:

```cpp
template <typename T>
void f(T x) {}

void g(auto x) {}
```

Заметим, что g эквивалентно f. А значит, что пакетные шаблоны тоже можно писать через auto:

```cpp
void g(auto&&... args) {}
```

У шаблонов есть [[Шаблоны|нетиповые параметры]] и в них тоже можно писать auto! (тоже с C++20)

```cpp
template <auto N>
struct Example {};

int main() {
	Example<2> ei;
	Example<'a'> ec;
}
```

Но все-таки поведение auto немного отличается от шаблонов.

```cpp
#include <initializer_list>

template <typename T>
void h(T) {}

int main() {
	auto lst = {1, 2, 3}; // OK
	h({1, 2, 3});         // CE 
}
```

В auto будет лежать `std::initializer_list`, а в f не найдет такую перегрузку. Выведет это:

```bash
test.cpp: In function ‘int main()’:
test.cpp:8:10: error: no matching function for call to ‘h(<brace-enclosed initializer list>)’
    8 |         h({1, 2, 3});         // CE
      |         ~^~~~~~~~~~~
test.cpp:4:6: note: candidate: ‘template<class T> void h(T)’
    4 | void h(T) {}
      |      ^
test.cpp:4:6: note:   template argument deduction/substitution failed:
test.cpp:8:10: note:   couldn’t deduce template parameter ‘T’
    8 |         h({1, 2, 3});         // CE
      |         ~^~~~~~~~~~~
```
## trailing return type
Все же auto был придуман не для того чтобы угадывать возвращаемое значение, а для того, чтобы сократить возвращаемый тип в функции или перенести его в конец, что повышает читаемость.

Так например `std::move` выглядит лучше в таком варианте:

```cpp
template <typename T>
auto move(T&& value) -> std::remove_reference_t<T>&& {
	return static_cast<std::remove_reference_t<T>&&>(value);
}
```

Нежели в таком:

```cpp
template <typename T>
std::remove_reference_t<T>&& move(T&& value) {
	return static_cast<std::remove_reference_t<T>&&>(value);
}
```

Стрелка после аргументов это синтаксическая конструкция позволяющая однозначно определить, какой возвращаемый тип у функции.

# decltype
Это выражение появившееся с C++11, которое позволяет вывести тип выражения на этапе компиляции (в отличии от typeid).

```cpp
#include <vector>

int main() {
	int x = 0;
	decltype(x) y = x; // int
	
	std::vector<decltype(y)> v; // std::vector<int>
}
```

В отличии от auto, decltype это не шаблонная подстановка, поэтому для dectype валидно выражение `std::vector<decltype(y)>`, а для auto - нет.
Кроме того, decltype не отбрасывает ссылки, он передает ровно то, что ему передали.

```cpp
int main() {
	int x = 0;
	int& y = x;
	decltype(y) z = y; // int&
}
```

Как мы знаем в рантайме невозможно отличить ссылку на объект от объекта и в аргументах функции тоже, а вот decltype это может.

Также на decltype можно навесить ссылки:

```cpp
int main() {
	int x = 0;
	int& y = x;
	decltype(x)&& t = x; // CE
	decltype(x)&& t = std::move(x); // OK
}
```

В примере показано, что decltype не имеет никакого отношения к универсальным ссылкам. Но правило [[(!)Типы выражений и ссылки|reference collapse]] работает

Чтобы проверить какой тип подставил decltype можно написать такую структуру:

```cpp
template <typename T>
struct Debug {
	Debug() = delete;
};

int main() {
	int x = 0;
	int& y = x;
	decltype(x)&& t = std::move(x); // OK
	
	Debug<decltype(t)>();
}
```

Будет ошибка компиляции с указанием, что нет конструктора по умолчанию для класса с шаблонным типом, который подставил decltype.

Рассмотрим еще один пример:

```cpp
#include <iostream>

int main() {
	int x = 0;
	
	decltype(++x) y = x;
	++y;
	std::cout << x << y;
}
```

Выведет 1 1. Важно понимать, что decltype вставляет тип на этапе компиляции, вычислений никаких на нем быть не может.
Еще один пример это подтверждает:

```cpp
int main() {
	decltype(throw 1)* v; // void*
}
```

Никакого исключения не будет.

Важно еще учитывать, что decltype по-разному работает с выражениями и переменными типов. Так, например, здесь будет вывод 1 1:

```cpp
#include <iostream>

int main() {
	int x = 0;

	decltype((x)) y = x; // int&, нетривиальный lvalue
	++y;
	std::cout << x << y;
}
```

Теперь посмотрим на decltype в возвращаемом тип функции:
Допустим хочется обращаться к контейнеру по индексу и с помощью функции менять его значение. Сразу же в голову приходит возвращаемый тип auto&

```cpp
#include <vector>

template <typename Container>
auto& get_elem(Container& c, size_t idx) {
	return c[idx];
}

int main() {
	std::vector<int> vi(5);
	get_elem(vi, 3) = 1; // работает
	
	std::vector<bool> vb(5);
	get_elem(vb, 3) = 1; // уже нет
}
```

Вот здесь и приходит decltype. Его использование возможно в 2 вариантах:

```cpp
template <typename Container>
auto get_elem(Container& c, size_t idx) -> decltype(c[idx]) {
	return c[idx];
}

template <typename Container>
decltype(auto) get_elem(Container& c, size_t idx) {
	return c[idx];
}
```

И теперь все скомпилируется.
Но почему нельзя было написать так:

```cpp
template <typename Container>
decltype(c[idx]) get_elem(Container& c, size_t idx) {
	return c[idx];
}
```

Просто потому что они еще не объявлены, когда компилятор их парсит.

Теперь разберемся с `decltype(auto)`. Ничего сложного: это инструкция компилятору выводить возвращаемый тип автоматически по правилам decltype, а не auto.


Возвращаясь к поведению decltype от разных выражений. Можно получить ситуацию когда всегда возврат функции будет ссылка, что провоцирует UB:

```cpp
template <typename Container>
decltype(auto) get_elem(Container& c, size_t idx) {
	decltype(auto) elem = c[idx];
	return (elem);
}
```

Возвращаемое значение из функции всегда будет битой ссылкой именно из-за работы decltype со скобками. Именно поэтому не стоит никогда писать выражение в скобках как возвращаемое значение.
