Цепочка выделения памяти на куче в контейнерах C++:

```
Container -> Allocator -> operator new -> malloc -> OS
```

Реализуем простейший `std::allocator`. Простейшие аллокаторы имеют только 4 метода.

```cpp
template <typename T>
struct allocator {
	T* allocate(size_t count) {
		return reinterpret_cast<T*>(new char[count * sizeof(T)]);
	}
	
	void dealocate(T* ptr) {
		delete[] reinterpret_cast<char*>(ptr);
	}
	
	template <typename U, typename... Args>
		void construct(U* ptr, const Args&... args) {
		new (ptr) U(args...);
	}
	
	void destroy(T* ptr) {
		ptr->~T();
	}
};
```

Также существует другие виды аллокаторов: `StackAllocator` и `PoolAllocator`.
StackAllocator способен получить копию аллокатора объекта созданного на стеке. Так, например, можно сделать список на стеке:

```cpp
int main() {
	std::array<char, 1000000> arr;
	StackAllocator<int> alloc(arr);
	std::list<int, StackAllocator<int>> lst;
}
```

Но такой простой аллокатор имеет недостатки. Например, в том же списке аллоцируется память не int, а типа Node*.

Методы construct и destroy были удалены из стандартного аллокатора с C++20

Аллокаторы имеют оператор равенства. Равенство аллокаторов возвращает true в том случае, если второй аллокатор может освободить память выделенную первым аллокатором.
# Allocator traits

В любом контейнере обращение за ресурсами проводится через allocator_traits и цепочка на самом деле выглядит так:

```
Container -> allocator_traits -> Allocator -> operator new -> malloc -> OS
```

Когда пишется свой аллокатор, то выясняется, что методы allocate, deallocate, construct и destroy реализуются одинаково. Поэтому была придумана общая обертка над аллокаторами, чтобы избежать написания лишнего кода.

Что из себя представляет allocator_traits:

```cpp
#include <iostream>

template <typename Alloc>
struct allocator_traits {
	template <typename U, typename...  Args>
	static void construct(Alloc& alloc, U* ptr, const Args&... args) {
		if constexpr (/*определен ли конструктор*/) {
			alloc.construct(ptr, args...);
		}
		else {
			new (ptr) U(args...);
		}
	}
	
	// похожим образом реализован destroy
};
```

Заметим, что методы allocator_traits рассматривают 2 варианта исхода: когда есть определенная функция в переданном аллокаторе (или конструктор у объекта) и когда нужно пройти по стандартному сценарию, как в std::allocator.

Кроме того, в allocator_traits лежат юзинги определенные в стандарте об используемых типах.

# Allocator Aware контейнеры

Когда контейнеры копируются не понятно, что нужно делать с аллокаторами. Не всегда он тривиальный и может быть такое, что при копировании новый аллокатор контейнера указывал на тот же пулл памяти, что и исходный. Для определения у аллокаторов опционально определяются методы: **select_on_container_copy_construction**, который определяет поведение при копировании контейнера с аллокатором. Возвращает тип аллокатора. Для остальных семантик определены похожие методы.
Также определена мета-функция **propagate_on_container_copy_assignment**, она возвращает std::true_type (что по факту является const bool) и если он true, то при присваивании контейнера другому, нужно присвоить аллокатор, а если false, то аллокатор не трогаем.
Любой Allocator-Aware контейнер обязан перед копированием и подобным обращаться к этим методам тоже через allocator_traits.

Так как методы являются опциональными, то allocator_traits доопределяет их, если не находит определение в своем шаблоне и по умолчанию возвращает копию.

В случае равенства аллокаторов есть оптимизация выделения памяти. В присваивании контейнера нужно проверить хватает ли памяти в текущем контейнере, чтобы уместить содержимое копируемого, если да, то дополнительно вызывать allocate для копирования не нужно, можно просто затереть объекты текущего и на месте них сконструировать копии второго. Для проверки равенства можно и нужно использовать мета-функцию allocator::is_always_equal, которая в компайл тайме определит нужно ли копирование.
Т.е. сначала проверяем равны ли аллокаторы, а потом, если не равны, копируем их.
