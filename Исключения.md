**Идея исключений**: безопасно обрабатывать ошибки при выполнении функции в рантайме. Раньше в С обработка ошибок была по возвращаемому значению из функции, а куда надо было положить результат передавалось по указателю в аргументы, после выполнения по коду ошибки проверяли, как выполнилась программа. Позже в C++ был придумал механизм исключений, который позволял досрочно завершить программу, если что-то идет не так. Например:

```cpp
#include <iostream>
#include <exception>

double divide(int a, int b) {
	if(b == 0) {
		throw std::logic_error("Divide by zero!");
	}
	return a/b;
}

int main() {
	divide(1, 0);
}
```

Что происходит, когда выбрасывается исключение: начинается процедура завершения программы, т.е. раскрутка стека вверх, что вызывает деструкторы в обратном порядке для всех локальных переменных на текущем уровне стека (уровни стека отделяются заходами в методы и функции, а также другими скоупами).
Есть и инструменты ловли исключений: `try {} catch() {}`

Добавим в наш код эту конструкцию:

```cpp
#include <iostream>
#include <exception>

double divide(int a, int b) {
	if(b == 0) {
		throw std::logic_error("Divide by zero!");
	}
	return a/b;
}

int main() {
	try {
		divide(1, 0);
	}
	catch(std::logic_error& err) {
		std::cout << err.what();
	}
}
```

Эта конструкция неразделима и используется всегда вместе. Как мы говорили раннее, когда выбрасывается исключение начинается раскрутка стека, в случае, если она видит скоуп с try, то раскрутка останавливается и отлавливаемое исключение передается в catch, НО это работает как передача аргументов в функцию, так что может быть копирование.

У **std::exception** есть полезный метод `what()`, который возвращает строку, в которой должно быть пользователем описано, что случилось.

Важно, что try catch не спасает от исключений, которые могут быть выкинуты выше по стеку, да и ниже тоже, а также от типов исключений, которые мы не ловим. Она работает только на своем скоупе, а в нем может быть сколько угодно уровней вниз.

В С раскрутка стека была бы вручную if-ми, мы бы проверяли не завершилась ли функция с ошибкой. А если бы мы были на 10 уровне стека, а обработать исключение могли бы только на 3, то огромная вереница из if-ов была бы необходима. С исключениями делается это очень просто и лаконично.

**Теперь о недостатках:**
try catch очень дорогая операция по процессорному времени, даже дороже new и delete вместе взятые.

# Использование исключений

Какие же стандартные функции выбрасывают исключения?

## Стандартные операторы выбрасывающие сключения
### new
Это оператор new (delete исключений не кидает, а иначе как)

```cpp
#include <iostream>

int main() {
	try {
		new int[1'000'000'000'000];
	}
	catch(std::logic_error& err) {
		
	}
}
```

Мы увидим такой вывод:

```txt
terminate called after throwing an instance of 'std::bad_alloc'
  what():  std::bad_alloc
Аварийный останов (образ памяти сброшен на диск)
```

В данном случае выкинула исключение std::bad_alloc стандартная функция std::terminate. Эта функция вызывается по многим сценариям некорректной работы программы и один из них это выброс не пойманного исключения. Функция служит для вывода сообщения, что в программе что-то пошло не так, она следит за внутренним состоянием выполнения, т.к. вшита в компилятор. После нее вызывается С функция `abort()`, которая просит операционную систему убить процесс с вердиктом **aborted**.

### dynamic_cast
Выкидывает исключение, по скольку этот инструмент нужен для рантайм операций. Иначе не понятно как ему сигнализировать об ошибке, когда недопустимо кастуют к ссылке. Выкидывается исключение std::bad_cast.

### typeid
Выкидывает исключение при передаче nullptr.

### throw
Да, оператор который создан, чтобы бросать исключения их кидает.

## Непосредственное использование

Рассмотрим пример:

```cpp
#include <iostream>

struct A {
	A() { std::cout << "A\n"; }
	~A() { std::cout << "~A\n"; }
	A(const A&) { std::cout << "copy\n"; }
}; 

void f(int x) {
	A a;
	if(x == 0) {
		throw a;
	}
}

int main() {
	try {
		f(0);
	}
	catch(...) {
		std::cout << "caught!\n";
	}
}
```

Что произойдет когда мы заходим выбросить как исключение объект A? И если выбрасывается объект, который обрабатывается в рантайме, где он будет храниться?
При выбрасывании исключения мы неизбежно будем копировать все объекты переданные throw, просто потому что они локальные и уничтожатся после выхода из скоупа. А храниться объект будет не на стеке, а на куче (не документировано в стандарте), просто иначе никак.
В процессе работы мы сначала вызовем конструктор A, потом скопируем его, потом вызовем деструктор для локального А, а потом поймаем исключение и только после выхода их catch вызовем деструктор копии.

Output:
```txt
A
copy
~A
caught!
~A
```

А если мы будем ловить непосредственно объект А, то копирований будет больше:

```cpp
#include <iostream>

struct A {
	A() { std::cout << "A\n"; }
	~A() { std::cout << "~A\n"; }
	A(const A&) { std::cout << "copy\n"; }
}; 

void f(int x) {
	A a;
	if(x == 0) {
		throw a;
	}
}

int main() {
	try {
		f(0);
	}
	catch(A a) {
		std::cout << "caught!\n";
	}
}
```

Output:
```txt
A
copy
~A
copy
caught!
~A
~A
```

Копирование происходит при передаче объекта в catch и копируем мы его уже на стек, при этом объект на куче никуда не денется. Исправлением данной ситуации будет принимать на вход исключение по ссылке, так мы будем обращаться к объекту на куче.

Также catch позволяет пробросить исключение дальше с помощью внутреннего throw:

```cpp
try {}
catch (...) {
	throw;
}
```

Без аргументов это пробросит дальше то, что поймает в catch, а если захотим выбросить что-то новое, то скопирует.

Если же исключение не ловить, то функция std::terminated, просто выведет результат вызова what(), если он есть в переданном исключении, если нет, скажет тип выброшенного объекта. К тому же, когда выбрасывается неотлавливаемое исключение компилятор не гарантирует, что после этого вызовется деструктор выброшенного объекта. Компилятор в этом случае только гарантирует, что процесс будет убит.
### multiple catch
Рассмотрим пример:

```cpp
#include <iostream>

int main() {
	try {
		throw 1;
	}
	catch (double d) {
		std::cout << "double\n";
	}
	catch (long long l) {
		std::cout << "long long\n";
	}
	catch (...) {
		std::cout << "other\n";
	}
}
```
Output:
```txt
other
```

```cpp
#include <iostream>

int main() {
	try {
		throw 1;
	}
	catch (double d) {
		std::cout << "double\n";
	}
	catch (long long l) {
		std::cout << "long long\n";
	}
}
```
Output:
```txt
terminate called after throwing an instance of 'int'
```

Как мы видим из примера для исключений не работают правила перегрузки. Выбирается один единственный подходящий вариант, никаких приведений типов нет.
Но ловятся по ссылке на родителя наследники, причем выбирается первый объявленный, и есть конверсия константных исключений.

Посмотрим на примеры с наследованием подробнее:
```cpp
#include <iostream>
struct Mom {};
struct Son: Mom {};

int main() {
	try {
		Son s;
		throw s;
	}
	catch (Mom) {
		std::cout << "caught Mom\n";
	}
	catch (Son) {
		std::cout << "caught Son\n";
	}
}
```

Здесь компилятор укажет, что вообще-то поймать базовый класс будет достаточно и catch наследника будет всегда пропущен:

```txt
test.cpp: In function ‘int main()’:
test.cpp:13:9: warning: exception of type ‘Son’ will be caught by earlier handler [-Wexceptions]
   13 |         catch (Son) {
      |         ^~~~~
test.cpp:10:9: note: for type ‘Mom’
   10 |         catch (Mom) {
      |         ^~~~~
```

Если же наследника сделать приватным, то поймаем только сына.

Теперь же рассмотрим ромбовидное наследование:

```cpp
#include <iostream>
struct Granny {};
struct Dad: Granny {};
struct Mom: Granny {};
struct Son: Mom, Dad {};

int main() {
	try {
		Son s;
		throw s;
	}
	catch (Granny) {
		std::cout << "caught Granny\n";
	}
	catch (Son) {
		std::cout << "caught Son\n";
	}
}
```

Здесь поймаем Son, ибо непонятно чей он наследник.
Если же добавим виртуальное наследование, то будет то же предупреждение, что и в первом случае:

```cpp
#include <iostream>
struct Granny {};
struct Dad: virtual Granny {};
struct Mom: virtual Granny {};
struct Son: Mom, Dad {};

int main() {
	try {
		Son s;
		throw s;
	}
	catch (Granny) {
		std::cout << "caught Granny\n";
	}
	catch (Son) {
		std::cout << "caught Son\n";
	}
}
```
Output:
```txt
test.cpp: In function ‘int main()’:
test.cpp:15:9: warning: exception of type ‘Son’ will be caught by earlier handler [-Wexceptions]
   15 |         catch (Son) {
      |         ^~~~~
test.cpp:12:9: note: for type ‘Granny’
   12 |         catch (Granny) {
      |         ^~~~~
```

### Исключения в конструкторе

```cpp
#include <iostream>

struct A {
	A() { std::cout << "A" << std::endl; }
	~A() { std::cout << "~A" << std::endl; }
};

struct B {
	A a;
	B(int x) {
		std::cout << "B" << std::endl;
		if(x == 0) {
			throw 1;
		}
	}
	~B(){
		std::cout << "~B" << std::endl;
	}
};

int main() {
	try {
		B(0);
	}
		catch(...) {
	}
}
```

Здесь логично предположить, что вызовется конструктор и деструктор А, а в В только конструктор. И это действительно так, поскольку сам объект В не был сконструирован, значит и деструктор не от чего вызывать.

А если бы мы захватывали бы какой-то ресурс в объекте, то была бы вызвана утечка памяти:

```cpp
#include <iostream>

struct A {
	A() { std::cout << "A" << std::endl; }
	~A() { std::cout << "~A" << std::endl; }
};

struct B {
	A* a;
	B(int x): a(new A()) {
		std::cout << "B" << std::endl;
		if(x == 0) {
			throw 1;
		}
	}
	~B(){
		std::cout << "~B" << std::endl;
		delete a;
	}
};

int main() {
	try {
		B(0);
	}
		catch(...) {
	}
}
```

Поэтому любое поле класса, который имеет исключения в конструкторе и не только, должен иметь [[RAII|смарт-обертку]] над полями захватывающими ресурсы.

Хорошо, с этим разобрались, но есть же еще списки инициализации. Представим ситуацию:

```cpp
#include <iostream>

struct A {
	A(int x) { 
		std::cout << "A" << std::endl;
		if(x == 0) throw 1;
	}
	~A() { std::cout << "~A" << std::endl; }
};

struct B {
	A a;
	A aa;
	A aaa;
	B(int x): a(1), aa(0), aaa(2) {
		std::cout << "B" << std::endl;
	}
	~B(){
		std::cout << "~B" << std::endl;
		delete a;
	}
};

int main() {
	try {
		B(0);
	}
		catch(...) {
	}
}
```

Чтобы обрабатывать исключения на уровне конструктора В есть возможность использовать try catch для списков инициализации. Называется это **function-try блок**:

```cpp
#include <iostream>

struct A {
	A(int x) { 
		std::cout << "A" << std::endl;
		if(x == 0) throw 1;
	}
	~A() { std::cout << "~A" << std::endl; }
};

struct B {
	A a;
	A aa;
	A aaa;
	B(int x) try: a(1), aa(0), aaa(2) {
		std::cout << "B" << std::endl;
	}
	catch (...) {
		std::cout << "caught!" << std::endl;
	}
	
	~B(){ 
		std::cout << "~B" << std::endl;
		delete a;
	}
};

int main() {
	try {
		B(0);
	}
		catch(...) {
	}
}
```

При таком подходе у всех объектов созданных до исключения вызовется деструктор, у остальных вызывать нечего.

Важно, что мы после вызова конструктора с исключением мы не знаем в каком состоянии созданный объект и поэтому в catch исключение всегда неявно пробрасывается дальше.

Также function-try блок работает и для функций, если хочется ее полностью поместить под try:

```cpp
void f() try {
	// ...
}
catch(...) {
	// ...
}
```

### Исключение в деструкторе
Бросать исключение в деструкторе категорически нельзя. Поскольку, когда бросается исключение, то предполагается, что вызов деструктора будет корректным. Иначе просто бесконечный цикл деструкторов и исключений.
Но обойти этот запрет можно:

```cpp
struct A {
	~A() noexcept(false) {
		throw 1;
	}
};
```

Таким образом, исключение деструкторе всегда будет вызывать std::terminated, т.к. летит 2 исключения подряд.

### Проверка исключения
```cpp
struct A {
	~A() noexcept(false) {
		if(!std::uncaught_exception()) {
			throw 1;
		}
	}
};
```

Встроенная функция `std::uncaught_exception` позволяет определить выброшено ли исключение в рантайм или нет (возвращает bool). Переопределить ее нельзя, она вшита в компилятор.

Но эта функция устарела и теперь используется `std::uncaght_exceptions()`, эта функция позволяет определить сколько исключений летит при вызове и еще не поймано.
Ситуация такая: пусть где-то выкинуло исключение и мы полезли в деструктор класса. Там вызывается функция, которая тоже выкинула исключение и мы его поймали. Теперь в рантайме у нас летит 2 исключения подряд.


# Exception safety
Когда предполагается, что код может кидать исключения или шаблонный класс принимает тип, который кидает исключения, то хочется написать его безопасным относительно исключений.
В стандарте разделены виды безопасности относительно исключений:
- Исключений нет вообще
- Нет гарантий. Выбрасывается исключение <=> UB.
- Базовая защита, по крайней мере все объекты остаются валидными. Не обязательно будет возвращено к исходному состоянию. Не будет UB и утечек памяти.
- Сильная защита, если кто-то бросает исключение, то функции возвращают все как было до вызова.
## noexcept
Это обещание компилятору и пользователю, что **только из этой функции** никакие исключения не будут выброшены или все что брошено на этом же уровне стека будет поймано здесь же.
Если же все-таки внутри выбрасывается исключение, то это приводит сразу же в std::terminated. 

Noexcept бывает еще условный, то есть noexcept принимает в аргументы определенное выражение, которое возвращает true/false на этапе компиляции, и определяет характеристику относительно исключений для функции.

Пример:

```cpp
template <typename T>
void f() noexcept(std::is_reference_v<T>) {

}
```

Если тип Т это ссылка, то я гарантирую безопасность относительно исключений.

Также иногда хочется проверять безопасность относительно исключений в зависимости от другой функции:

```cpp
template <typename T>
void g() {}

template <typename T>
void f() noexcept {
	noexcept(g<T>());
}
```

Здесь `noexcept()` это оператор, который проверяет условие на этапе компиляции.
Но хочется все-таки в условие поставить:

```cpp
template <typename T>
void g() {}

template <typename T>
void f() noexcept(noexcept(g<T>())) {}
```

Теперь функция f зависит от безопасности исключений относительно g.

На практике noexcept ставится там, где вообще ничего плохого произойти не может. Так например оператор [] не noexcept.
# Понятие исключений

Не все рантайм ошибки это исключения.

Например segfault это не исключение, а именно рантайм ошибка. Его не получится поймать.

```cpp
#include <iostream>
#include <vector>

int main {
	std::vector<int> v;
	try {
		v[1000000] = 1;
	}
	catch(...) {
		std::cerr "Caught!\n";
	}
}
```

Здесь используется многоточие в catch и оно означает, что мы будем выполнять внутри него код, если хотя бы какое-то исключение выкинуто. Работает как пакет.

Также деление на 0 это тоже ошибка, а не исключение:

```cpp
#include <iostream>

int main {
	try {
		double d = 1 / 0;
	}
	catch(...) {
		std::cerr "Caught!\n";
	}
}
```

Вообще, это тоже исключение, но на уровне процессора и средствами C++ оно не ловится.

**aborted** это тоже ошибка, а не исключение.
**stackoverflow** этот тоже ошибка, а не исключение.

## Иерархия стандартных исключений

Есть главный родительский класс **std::exception**. Он полиморфный и имеет виртуальный деструктор и виртуальный метод **what()**.

Глобально все исключения разделяются на 2 вида: **logic_error** и **runtime_error**.

Здесь runtime_error никаким образом не связано с segfault-ом и другими неуловимыми райнтайм ошибками.

## logic_error
Выкидывается исключение такого типа, если в этом виноват сам пользователь.
Это как правило вызывается от неправильных переданных аргументов.
Разделяется на подтипы: **invalid_error**, **domain_error**, **lenght_error**, **out_of_range**, **future_error**.
## runtime_error
Выкидывается исключение такого типа, если эта ошибка не зависит от пользователя, в данном случае действительно что-то пошло не так.
Разделяется на подтипы: **range_error**, **overflow_error**, **system_error** и другие.
